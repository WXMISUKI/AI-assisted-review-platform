# 开工条件核查投产路线图与决策沉淀

更新时间：2026-07-23

## 一句话结论

下一阶段最值得推进的方向是：**把开工条件核查从“能跑通试点”推进到“可重复验收、可交付、可回溯的单项目试点交付包”**。

不要优先转向完整多租户权限、完整数据库迁移、更多 provider 能力或局部 UI 美化。当前最值钱的是让真实资料链路稳定：资料接入、正式核查、人工复核、报告生成、归档、下一轮整改复审，每一步都可解释、可追踪、可复验。

## 当前已形成的判断

### 1. 平台定位

本项目不是单纯的“工程大模型问答平台”，也不是把 Dify 工作流搬进页面。

更准确的定位是：

```text
工程资料版本审查台账 + AI 辅助核查平台 + 智能资产沉淀平台
```

其中：

- 平台 own 项目、任务、状态、版本、人工结论、报告、历史留痕。
- OCR / MaxKB / LLM / Dify 类能力是 provider 或智能能力，不应 own 业务状态。
- AI 输出只能作为证据、建议、检索摘要或辅助报告草稿，不能自动替代施工单位、监理单位和相关责任人的最终审核责任。

### 2. 成熟平台启发

来自 Procore、Autodesk Construction Cloud、Oracle Aconex 等成熟工程资料平台的核心启发：

- 强平台不是只做“单次审查”，而是做版本化复审。
- 对用户暴露的是清晰交付状态，不是内部状态字段。
- `Next action by`、`Next action due`、当前责任人、超期、卡点，是一线平台的一等能力。
- 历史版本、归档记录、审批流、报告资产和审计轨迹必须保留。

来自云工大模型这类建筑行业 AI 平台的启发：

- AI 能力需要平台化、模块化、资产化。
- 不能只堆一个模型入口，而要沉淀知识库、规则、智能体、提示词、模板、评测集和运行记录。
- 业务闭环和智能资产不是二选一：先让业务作业流站稳，再逐步把 AI 能力资产化。

## 已完成的关键基础

当前项目已经具备以下基础：

- 单项目真实试点链路：资料接入、正式核查、人工复核、报告生成、归档。
- 任务状态机：`packet_uploaded`、`awaiting_human_review`、`report_ready`、`archived` 等。
- 归档只读约束：归档 run 不允许重新正式核查、重新生成报告或重新初始化。
- 下一轮整改复审：归档后用新的 run-specific task id 创建新轮次，不污染旧 run。
- checklist matrix：按核查项展示匹配、缺失、原因、证据和人工复核关联。
- report workbench：已开始展示报告摘要、问题清单、历史轮次和整改对照。
- 后端验收 smoke：`npm run smoke:opening-condition` 可验证核心链路。
- OpenSpec 主规格已沉淀：状态、复审、报告、责任队列、项目对象模型、acceptance smoke 等。

## 最近一次 smoke 固化的重要规则

`npm run smoke:opening-condition` 固化了以下投产关键规则：

- 有 open/deferred 人工复核项时，报告生成必须阻塞。
- 人工复核阻塞项全部处理后，任务才能进入 `report_ready`。
- 报告生成后才允许归档。
- 归档后 task 与 report asset 都进入 archived。
- archived run 不允许再执行正式核查、报告生成或重新初始化。
- 下一轮复审必须创建新的 run-specific task id。
- 新 run 不能偷用旧 archived run 的临时内嵌状态，必须依赖 workspace 级正式依据、主数据和 ready 知识库。

这条规则非常关键：它把“历史留痕不被污染”和“整改复审可重复运行”落到了可自动验证的工程约束上。

## 下一阶段优先级排序

### P0：链路验收从后端 smoke 扩到 HTTP / 页面级

目标：不再只靠人工点击发现链路问题。

任务拆分：

1. 增加 HTTP smoke，覆盖 `/api/opening-condition/pilot-tasks/*` 的关键接口链路。
2. 增加轻量页面 smoke，验证按钮启禁用、归档只读、下一轮复审入口不误触。
3. 将 smoke 输出变成试点验收清单：当前 run id、状态、阻塞原因、报告状态、归档状态、历史轮次。
4. 在 runbook 中写明每次真实样本联调前后必须跑的命令和预期。

价值：这是最短路径降低“页面看起来能点，但链路实际错”的风险。

### P1：报告交付语义增强

目标：让报告页从“平台内部摘要”走向“监理可读的交付件”。

任务拆分：

1. 按核查项输出：核查项目、所属分类、是否必查、匹配文件、缺失原因、风险等级、整改建议。
2. 对齐 Dify 工作流报告中的“不符合项清单”表达，但报告事实必须来自平台 task/checkItems/humanReview/reportAsset。
3. 将 `fail`、`blocked`、`needs_human_review`、`rejected` 等内部状态映射成面向操作者的交付语义。
4. 把历史轮次详情做成可进入查看的完整报告，而不仅是摘要列表。
5. 保留后续导出 docx/pdf 的扩展点，但短期先完善页面报告结构。

价值：这是试点能否给监理和施工方看懂、愿意继续用的关键。

### P2：复审入口和历史轮次收口

目标：让“重新上传资料开始下一轮”只有一个明确入口，旧轮次始终只读。

任务拆分：

1. 报告页作为“发起下一轮整改复审”的主入口。
2. 资料接入页只在明确 rerun 模式下展示新轮上传动作。
3. 历史轮次只允许查看详情和报告，不暴露会改变当前任务的动作。
4. 当前 run、历史 run、archived run、rerun draft 的 UI 文案和按钮状态统一。

价值：减少误操作，避免出现“我点了哪里进入第四轮”的混乱。

### P3：依据入库预览与发布治理

目标：让用户先看清楚平台准备入库什么，再允许其成为正式核查依据。

任务拆分：

1. 上传合同/资质依据后生成结构化预览：主体、合同包、资质边界、人员设备范围、有效期、来源文件。
2. 人工确认后发布为 workspace 级正式依据版本。
3. 主数据同理：候选识别 -> 人工确认 -> 发布 -> 绑定 run。
4. 新 run 只能绑定已发布依据、已确认主数据和 ready 知识库。

价值：解决“这些确认是什么意思”“是不是资料包里不存在”的认知混乱。

### P4：前端工作台体验重构

目标：在链路稳定后，把拥挤的页面整理成成熟平台式工作台。

任务拆分：

1. 工作区总览减少密度，采用“当前任务 / 下一动作 / 风险摘要 / 历史轮次”分区。
2. 资料核查矩阵优先表格化，支持按缺失、待复核、已匹配筛选。
3. 人工复核页按“阻塞报告 / 可延期 / 已处理”分组。
4. 报告页使用更强的层级和问题清单视觉，不再让每个列表项看起来一样。

价值：提升一线用户理解效率，但必须排在链路正确性之后。

### P5：智能资产管理

目标：让平台越来越聪明，而不是每次只跑一次模型。

任务拆分：

1. 知识库资产：按项目、审查对象、参建主体、合同包隔离。
2. 智能体资产：区分资料完整性核查、合规性核查、报告生成、整改复核等 agent。
3. 规则资产：沉淀 deterministic matching 规则、风险等级规则、整改建议模板。
4. 提示词和模板资产：版本化管理，不把 prompt 散落在页面或临时代码里。
5. 评测资产：把真实试点样本的输入、期望输出、误判点沉淀成 regression set。

价值：这是长期壁垒，但要建立在 P0-P3 的稳定作业流之上。

## 当前明确不做

- 不先做完整多租户权限平台。
- 不先做完整数据库迁移。
- 不继续在工作区总览做局部 UI 无限优化。
- 不让前端直连 OCR / MaxKB。
- 不把 provider 命中结果直接当作正式审核结论。
- 不为了某个文件识别准确率无限调试 provider。

## 推荐下一组 OpenSpec 方向

下一组 change 建议命名：

```text
opening-condition-http-ui-smoke-and-report-handoff
```

目标：

- 在已有后端 domain smoke 基础上，补 HTTP/页面级验收护栏。
- 把报告页交付语义进一步产品化。
- 明确真实样本试点前后如何验收。

这组完成后，再进入“依据入库预览与发布治理”和“前端工作台体验重构”会更稳。
## 2026-07-23 补充：本轮验收护栏完成后的排序

`opening-condition-http-ui-smoke-and-report-handoff` 完成后，P0 的重点不再是继续加更多按钮，而是先把三层护栏跑顺：

1. `npm run smoke:opening-condition`：守住后端领域状态机。
2. `npm run smoke:opening-condition:http`：守住公开 API 链路和 archived 不可变。
3. `npm run smoke:opening-condition:ui`：守住 archived 只读、历史详情只读、报告页交付语义这些前端边界。

如果三层 smoke 都通过，下一阶段优先进入 `依据入库预览与发布治理`：上传合同/资质依据后先展示平台准备入库的结构化事实，由人工确认后发布为正式依据和主数据。只有当依据治理足够清楚后，再进入更大范围的前端体验重构；否则 UI 做得再漂亮，也仍然解释不清“平台到底把什么当成正式审核依据”。

前端体验重构仍保留在路线图内，但定位为 P4：等链路验收和依据治理稳定后，再基于已有成熟平台调研与设计系统统一重做工作区密度、报告列表层级、状态色和组件拆分。
