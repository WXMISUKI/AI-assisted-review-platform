# 开工条件真实试点：人工复核到报告归档交付说明

更新时间：2026-07-20

## 当前结论

当页面提示“正式核查已完成，进入人工复核 · 待处理 N 项”时，说明真实资料已经完成以下链路：

```text
浏览器上传真实文件
-> 对象引用进入后端试点任务
-> ZIP manifest 资料清单提取
-> 知识库门禁绑定
-> 正式资料核查
-> 生成人工复核队列
```

这表示“文件上传到识别/核查”的主链路已经走通。下一步应进入人工复核，而不是继续调上传或重复初始化任务。

## 人工复核按钮含义

- 确认通过：监理或试点操作员确认该项证据满足要求，关闭阻塞。
- 修正后通过：资料本身可接受，但平台识别、归类或说明需要人工修正，关闭阻塞并在报告中计入修正。
- 延期补件：当前资料不足或现场需补充，仍保持阻塞，不能生成报告。
- 驳回不通过：确认该项不符合要求，关闭阻塞，后续报告应体现不通过或风险结论。

只有 open/deferred 复核项全部关闭后，任务才应进入 `report_ready`。报告页在任务未进入 `report_ready` 前不应启用“生成报告摘要”。

## 与成熟工程资料审核平台的对应关系

成熟平台通常围绕提交项、资料包、审查流、审查响应、分发/关闭和审计记录组织流程。自动化可以辅助归集、初筛、检索和提醒，但正式审查结论仍需要人审记录承载。

本项目当前链路与这个方向一致：

- 平台持有项目/合同包/参建单位/资料包/任务状态。
- OCR、知识库和模型只作为 provider 支撑，不直接给最终审批结论。
- 正式核查将缺失、低置信、签章签名勾选等不确定点路由到人工复核。
- 人工复核关闭阻塞后再生成报告并归档。

## 下一阶段优先级

最高价值方向是继续固化“人工复核到报告归档交付包”，而不是先做完整多租户权限、完整数据库迁移或继续局部 UI 打磨。

推荐任务组：

1. 人工复核交付固化
   - 明确每个复核项的处理含义。
   - 显示阻塞、已关闭、延期等计数。
   - 保证延期项继续阻塞报告。
   - 保证驳回项进入报告结论，而不是阻塞报告生成。

2. 报告归档门禁固化
   - 只有 `report_ready` 且无 open/deferred 项时启用报告生成。
   - 报告包记录输入资料、核查数量、证据数量、复核决策数量和归档状态。
   - archived 后不允许再次生成或修改报告。

4. 人工复核决策台账
   - 报告包保留每个复核项的最终状态、原因、审核人和时间戳。
   - 台账只保留安全摘要，不暴露原始 OCR、私有链接或长文本。
   - archived 报告应能回看这批决策，而不是只看汇总数字。

5. 真实样本问题台账
   - 记录 3 项人工复核的业务原因。
   - 区分“资料确实缺失”和“平台识别/归类需修正”。
   - 只记录阻塞真实交付的问题，避免陷入识别准确率无限调优。
