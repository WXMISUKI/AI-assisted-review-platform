# 开工条件核查下一阶段开发计划

更新时间：2026-07-20

## 推荐方向

下一阶段优先做“单项目真实试点闭环”，而不是先建设完整多租户权限平台、继续打磨局部 UI、或者进一步深挖 MaxKB/OCR provider 内部能力。

推荐闭环如下：

```text
平台任务初始化
-> 合同/依据、核查表、资料包对象引用落入试点任务
-> ZIP manifest 提取
-> 正式资料匹配
-> 人工复核不确定项
-> 生成内部辅助报告摘要
-> 归档任务和报告资产
```

## 为什么这是最高价值切片

- 它最能验证项目定位：平台是业务事实和审查结论的 owner，OCR Worker 与 MaxKB 是外部能力 provider。
- 它最接近真实投产路径：用户能从资料上传一路走到报告归档，而不是停留在连通性或 demo 展示。
- 它最快暴露风险：核查表适配、资料包命名混乱、人员设备主数据授权、签章/勾选人工复核、报告归档状态都会在一个链路里暴露。
- 它避免过早建设重平台：多租户、细权限、正式数据库迁移、复杂队列都应该在试点闭环跑通后再按痛点升级。

## 任务组拆分

### A. 试点完成闭环

- 将后端 `checkItems`、`evidence`、`humanReviewQueue`、`reportAsset` 作为前端优先展示来源。
- 人工复核支持确认、修正、延期、驳回，并刷新任务状态。
- 报告生成和归档由操作员显式触发。

### B. 真实联调样本

- 使用一个真实合同/依据文件、一个核查表和一个资料 ZIP 包跑完试点。
- 记录每一步的阻塞原因、误判原因和人工复核项。
- 将无法由 OCR 稳定识别的签章、签名、勾选、日期作为人工复核项，而不是让模型强行给最终结论。

### C. Provider 对接后续

- 前置平台继续负责 ZIP 拆包、文件清单、资料类型初判、业务状态和人工结论。
- OCR Worker 负责 OCR、结构化后处理、可控入库和 provider 检索验收。
- MaxKB provider 负责安全检索和知识库支撑，不承载项目、队伍、任务或审批状态。
- OCR/Provider 的结构化输出先进入平台 owned 的 basis preview，保留 provenance、missingFields、confidence 和 nextAction。
- provider preview 只能作为待确认预览，不得绕过人工确认直接发布为正式依据。

### D. 后续升级顺序

1. 跑通单项目真实试点闭环。
2. 收口统一状态门禁与只读历史入口，避免 archived/当前 run/复审上传逻辑散落在各页。
3. 补项目 / 工作区 / 参与主体的资产注册与隔离视图，让依据、主数据、知识库和历史轮次先按上下文收口，再扩试点范围。
4. 补统一运行快照事实层，收口“当前轮 / 历史轮 / 整改复审入口 / 待人工判断”语义，避免报告页和归档页继续各算各的。
5. 补正式数据库或轻量持久化迁移方案。
6. 补任务队列和长任务状态恢复。
7. 补组织/项目/队伍权限模型。
8. 再扩展更多资料类型、OCR 后处理规则和知识库检索质量评估。

## 新增说明：为什么现在要优先做资产隔离

- 真实试点一旦从单项目扩到“一个监理管多个项目、一个项目有多个危大工程、一个危大工程有多个参建主体”，最先出问题的不是模型，而是上下文混用。
- 成熟平台真正稳定的地方，在于让操作员一眼看清“我现在在哪个项目、哪个对象、哪个主体、绑定了哪套依据、哪套主数据、哪个知识库、历史跑过几轮”。
- 所以下一个最有价值的大方向，不是继续补局部页面效果，而是把工作区资产注册、状态归属和历史轮次隔离先做成统一事实层。

## 新增说明：为什么资产隔离后要补运行快照内核

- 资产边界看清之后，下一步最容易出错的就是“当前到底看的是哪一轮、这轮还能不能动、下一轮该从哪里发起”。
- 如果报告页、归档页、历史轮次和整改复审入口各自维护一套状态语义，试点越跑越乱，用户会不断怀疑系统结论是否可靠。
- 所以在真正上更多项目和更多监理使用前，需要先把运行快照、只读历史和整改复审入口统一成一个共享事实层。

## 当前不做

- 不做完整多租户权限平台。
- 不让前端直连 MaxKB 或 OCR Worker。
- 不新增第二套上传通道。
- 不把模型或 provider 命中结果直接当正式审查结论。
- 不为了局部识别准确率无限调试 provider。
