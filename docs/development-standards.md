# 企业级开发规范

这份文档是本项目前后端开发的默认遵循标准。凡是会影响数据结构、状态流转、接口契约、权限、安全、样式系统或交付行为的改动，优先按这里执行，再进入具体实现。

## 1. 核心原则

- 先规格，后实现，再归档。
- 任何会改变业务行为的改动，先写 `openspec` 变更，再改代码。
- 任何会改变状态机、数据结构、接口字段、权限或安全策略的改动，都必须同步更新文档。
- 变更范围保持最小，只改完成任务所需的边界。
- 不把临时调试代码、一次性脚本、验证文件留在正式仓库里。

## 2. 仓库组织

- `src/domain` 放业务模型、状态流转、任务服务、纯逻辑工具。
- `src/*Page.tsx` 或等价文件放页面级组件，保持薄入口、厚领域。
- `src/shared` 或等价目录放跨页面可复用组件、hooks、工具、类型。
- `docs/` 放产品、架构、运维、部署和开发规范文档。
- `openspec/` 放需求变更、规格和归档记录。

不建议：

- 把页面、状态逻辑、请求调用、格式化工具全部堆在一个大文件里。
- 在组件里直接拼接大量业务流程。
- 在多个地方复制同一份状态标签、错误文案、主题 token 或枚举映射。

## 3. 前端规范

### 3.1 React 组件

- 使用函数组件和 hooks。
- hooks 必须放在组件顶部，不能条件调用。
- 渲染阶段保持纯净，不做网络请求、写存储、改全局状态等副作用。
- 页面组件只负责编排，复杂计算抽到纯函数或领域模块。
- 组件参数保持明确，优先使用命名 props。
- 不要让一个组件同时承担视图、状态、数据转换和请求逻辑。

### 3.2 TypeScript

- `strict` 保持开启。
- 能写明确类型就不要依赖隐式推断。
- 不要随意使用 `any`；必须接外部不稳定数据时，先做类型收敛或运行时校验。
- 对外导出的函数、对象、枚举式常量，优先补齐类型。
- 复杂状态建议用领域类型表达，不在组件内部反复发明临时结构。

### 3.3 样式与主题

- 颜色、边框、阴影、文字颜色优先走设计 token 或 CSS 变量。
- 深浅主题必须成对检查，不能只保证浅色正常。
- 不要把组件默认白底、黑字当成最终结果；新组件必须检查暗色主题。
- 需要统一的交互状态，优先复用现有状态类名和 token。
- 长文本、长文件名、长标签必须做截断和 hover 全量展示。
- 重要可交互元素必须有 hover、focus、active、disabled 状态。

### 3.4 交互与可访问性

- 关键操作必须有明确按钮文案和确认态。
- 键盘操作不能被破坏。
- 表单元素需要可聚焦、可读、可恢复。
- 弹层、浮层、对话框需要处理视口边界和遮挡问题。
- 图标按钮必须有可理解的 aria 或 tooltip 语义。

### 3.5 状态管理

- 状态按归属放在最接近的层级。
- 页面状态不要跨层偷偷共享。
- 任务状态、文档状态、OCR 状态、流式进度分别建模，不要混成一个大对象后到处传。
- 需要跨页面复用的状态，优先放到服务层或领域层。

## 4. 后端规范

### 4.1 API 设计

- 一个接口只做一类业务。
- 请求和响应都要有明确 schema。
- 字段名保持稳定，避免前端靠猜。
- 返回错误要能被前端识别和展示，不要只给一个裸字符串。
- 异步任务必须有任务 ID、状态、进度和失败原因。

### 4.2 异步与流式

- OCR、审查、报告生成这类耗时任务必须走异步。
- 支持轮询、SSE 或 WebSocket 的统一事件模型。
- 流式事件要能恢复，不能只依赖一次性的前端内存状态。
- 用户离开页面后，后台任务要继续执行。

### 4.3 数据校验

- 上传文件、文档名、项目名、提示词内容、智能体配置都要做后端校验。
- 外部输入必须经过类型、长度、格式、空值校验。
- 处理中文文件名、空格、特殊字符和超长内容时要有明确策略。

### 4.4 日志与可观测性

- 日志要能区分业务日志、错误日志和调试日志。
- 不要把密钥、Token、完整用户隐私内容直接写进日志。
- 对任务流转、OCR、AI 调用、报告生成保留可追踪的状态点。

## 5. 安全规范

这部分建议长期以 [OWASP Top 10](https://owasp.org/www-project-top-ten/) 为底线。

- 不把密钥、Token、访问凭证提交到仓库。
- 环境变量只放在本地或安全配置中，前端只能读取公开环境变量。
- 文件上传要检查类型、大小、路径和存储位置。
- 任何外部 URL 读取都要考虑 SSRF 和非法跳转风险。
- 接口必须做权限判断，不能只靠前端隐藏菜单。
- 对角色、组织、任务可见范围做后端约束。

## 6. 配置与环境

- 本地开发配置与生产配置分离。
- 前端环境变量统一使用 Vite 的公开前缀，参考 [Vite 环境变量文档](https://vite.dev/guide/env-and-mode)。
- TypeScript、React 和模块边界的约束优先在代码层面落实，不能只靠约定。
- 新增依赖前先确认是否能被现有架构直接承接。

## 7. 代码质量与验证

- 每次改动先确认是否需要更新 spec、设计文档或接口契约。
- 默认只做必要的轻量验证，优先 `typecheck` 和针对性 smoke test。
- 只有重大改动才做更重的构建或全量测试。
- 临时验证脚本、一次性测试文件、调试产物，验证完立即删除。
- 若改动触及交互、状态流转或主题系统，优先做实际页面检查。

## 8. 版本控制与变更流程

- 大功能改动先走 `openspec` 变更。
- 实现按“规格 -> 实现 -> 归档”推进。
- 结构性调整尽量拆成小变更，不要一次把多个目标揉成一个大 PR。
- 分支命名保持可读、可追踪，建议带任务前缀。
- 提交信息保持明确，优先说明“做了什么”和“为什么做”。

## 9. 文档维护

- 架构、产品、开发规范、部署文档要彼此引用，而不是各写各的。
- 当状态流转、权限、页面结构、智能体角色变化时，更新文档。
- 归档后的 spec 应该能让后来的人快速理解当时为什么这么做。
