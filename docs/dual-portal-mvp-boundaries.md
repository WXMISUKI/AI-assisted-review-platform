# 双平台 MVP 边界与并行开发约定

更新时间：2026-07-24

## 产品定位

本项目是一个统一平台下的两个独立业务门户：

1. **施工方案审查**：面向施工方案文档，重点是 OCR/结构准备、问题识别、人工采纳或驳回、审查报告和处理后方案快照。
2. **开工条件核查**：面向项目/合同包/参建主体工作区，重点是依据与主数据发布、资料包完整性匹配、人工复核、整改复审、报告归档和 DOCX 交付。

两者都属于 AI 辅助、人工负责、可回溯的工程资料审查产品，但不能共享同一套业务状态机。

## 共用能力

- 统一登录、产品启动器和平台壳层。
- 主题 token、基础布局、按钮、状态展示基础样式。
- MinIO/对象存储、OCR、知识库、LLM、HTTP Tools 等 provider 适配器。
- 文档对象引用、报告交付 handoff、导出适配器契约。
- 有边界的诊断信息、活动记录和审计摘要。
- TypeScript 类型检查、通用开发规范和测试执行约定。

## 施工方案审查独立能力

- 文档任务和文档库。
- OCR 解析与结构恢复。
- 施工方案问题锚点、问题状态和人工处理。
- 审查报告资产。
- 修改模式下的处理后方案快照。

最小闭环：

```text
上传文档 -> OCR/结构准备 -> 生成问题 -> 人工接受/驳回/修改
-> 生成审查结果 -> 查看结果或处理后方案
```

## 开工条件核查独立能力

- Opening-Condition Workspace。
- 合同/资质依据版本和项目主数据。
- 资料核查表、资料包 manifest 和确定性匹配。
- 人工复核队列。
- 整改复审 run、历史轮次和归档只读。
- 开工条件报告资产与 DOCX 导出。

最小闭环：

```text
工作区 -> 发布依据/主数据 -> 上传核查表和资料包
-> 正式匹配 -> 人工复核 -> 报告生成 -> 归档
-> 新 run 整改复审
```

## 并行开发规则

- 施工方案改动默认只触及 `ReviewTask`、OCR、review issue、review result 相关模块。
- 开工条件改动默认只触及 opening-condition workspace、pilot task、checklist、human review、report/archive 相关模块。
- 共享模块只能放两条产品使用相同语义的能力，不能把某一个产品的业务对象塞进共享层。
- 大文件先抽取契约和纯函数，再按垂直切片拆分；禁止一次性重写核心闭环。
- 任何状态机、API 字段、主题 token 或跨产品导航变化，先更新 OpenSpec。

## 当前明确不纳入两个 MVP

- 完整多租户权限。
- 正式数据库迁移。
- 十二类问题全部独立智能体化。
- 全面 UI 重构。
- 复杂协同审批和消息中心。
