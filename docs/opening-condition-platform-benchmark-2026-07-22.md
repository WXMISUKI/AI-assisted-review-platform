# 开工条件核查平台对标调研纪要

更新时间：2026-07-22

## 调研目的

围绕“工程资料审核 / 版本复审 / 人工审批 / 报告留痕 / AI能力平台化”五个维度，对成熟平台与建筑行业大模型平台做官方资料对标，提炼适合当前项目下一阶段的高价值开发方向。

## 核心发现

### 1. 成熟资料审核平台的核心，不是“单次审核”，而是“版本化复审”

Procore 明确把 revision 当成一等公民：

- 只有最新 revision 才能继续创建新 revision。
- 新 revision 会自动继承上一版的大部分通用信息。
- revision number 自动按 `n + 1` 递增。
- 审批流、分发列表、包（package）和权限都围绕 revision 展开。

这说明：**真正成熟的平台不是覆盖旧结果，而是保留旧版、创建新一版、沿版本推进审批。**

### 2. 成熟平台对外只暴露清晰的交付状态，不暴露底层内部枚举

Autodesk Docs / Reviews 对外强调的是：

- `Approved`
- `Rejected`
- `In review`

而不是把内部中间态直接扔给用户。并且它还强调：

- 当前版本的 review status 单独可见
- 历史版本的 review status 可从 version history 查看
- 如果同一版本同时参与多个 review，当前状态仍可能保持 `In review`

这说明：**状态展示必须是面向操作者的最终语义，不是内部字段的直出。**

### 3. 成熟平台把“下一动作责任人”和“超期瓶颈”当成平台一级能力

Autodesk Reviews 里有：

- `Next action by`
- `Next action due`
- 当前 step
- review archive

Oracle Aconex 更进一步，把工作流做成可分析对象：

- workflow status
- workflow unique id
- workflow number
- overdue workflow reviews
- 哪个组织卡住了下一步

这说明：**平台不是只要知道“有问题”，还要知道“现在该谁处理、卡在哪、多久没动”。**

### 4. 云工大模型更像“建筑 AI 能力平台”，不是完整审查闭环平台

从上海建工官方资料看，云工大模型的优势主要在：

- `8层1体` AI 架构
- 多产品模块化能力
- 知识问答、施工计算、图纸、方案、材料质量文件检查等产品化工具

它对我们很有启发的地方主要是：

- AI 能力平台化
- 模型层、数据层、引擎层、应用层解耦
- 垂直场景拆成多个产品能力

但它不直接替代我们当前最缺的部分：**资料审核 run 的状态机、复审版本链、人工责任流、报告交付语义。**

## 对当前项目的启发

### A. 你们现在最缺的是“作业流产品化”，不是“AI能力更多”

当前项目已经有：

- 资料接入
- 正式核查
- 人工复核
- 报告归档
- 多轮 run

但还缺：

- 统一最终状态语义
- 唯一的复审主入口
- 明确的“谁下一步处理”
- 超期 / 卡点 / 当前责任人的平台表达

所以近期重点不该先放在：

- 再接更多模型
- 再扩更多 provider
- 再做更复杂的大模型能力

而应该放在：

- 让这条审核闭环真正可操作、可追责、可解释

### B. 你们的平台定位应更接近“工程资料版本审查台账 + AI辅助审查平台”

而不是单纯“问答型工程大模型平台”。

更准确地说：

- 云工大模型的启发在“能力平台化”
- Procore / Autodesk / Aconex 的启发在“审核工作流产品化”

你们当前更该优先向后者学习。

## 推荐的下一阶段开发优先级

### 第一优先级：审核状态与动作收口

目标：消灭状态冲突和重复入口，让操作员不会误解“这项到底处理完没、我下一步该去哪”。

包括：

- 统一 `check item`、`human review`、`report delivery` 三套状态
- 把“发起下一轮整改复审”做成唯一主入口
- 明确 archived run、current run、history run 的动作边界

### 第二优先级：责任人与时效驱动

目标：把“谁来处理、何时到期、卡在谁手里”做出来。

包括：

- 当前责任人
- 下一动作人
- 处理截止时间
- 超期提醒
- 按组织 / 队伍 / 专业分类的待办视图

### 第三优先级：项目 / 危大工程 / 分包队伍对象建模

目标：让平台从“单试点 run”走向“一个监理能真正管理多个项目多个对象”。

包括：

- 工作区 = 项目级容器
- 危大工程对象 = 审查主题
- 分包队伍 / 专业队伍 = 资质与资料审查主体
- 每个对象有独立的依据、主数据、知识库和 run 历史

### 第四优先级：依据入库预览与发布治理

目标：先看清楚“要入库什么”，再允许它成为正式依据。

包括：

- 依据识别预览
- 主数据识别预览
- 人工确认后发布
- 已发布版本绑定到某一轮 run

### 第五优先级：AI能力平台化

目标：学习云工大模型，把 AI 能力做成平台底座，而不是把业务状态丢给模型。

包括：

- OCR / 检索 / 规则 / 结构化抽取 / 生成总结分层
- provider 可替换
- 业务状态由平台 own，模型只负责能力输出

## 补充判断：流程产品化与智能资产化要双线推进

除了审核作业流本身，平台还需要持续沉淀“让平台越来越聪明”的资产层。这个判断是合理的，而且不应该被理解为和流程产品化二选一。

建议把平台后续设计拆成两条主线：

- `作业流产品化`：版本、状态、责任人、下一动作、历史留痕、整改闭环
- `智能资产化`：智能体资产、知识库资产、规则资产、提示词资产、模板资产、评测资产

其中优先级应当是：

1. 先把作业流产品化做到能稳定驱动试点
2. 再把智能资产化体系逐步抽出来，避免 AI 能力散落在页面和接口细节里

如果顺序反过来，平台会先变“聪明”，但不一定“能用”；而先把作业流站稳，再建设资产层，智能能力才有稳定的业务承载面。

## 结论

对当前项目最有价值、最能快速投产的下一阶段主方向，不是“继续加 AI 功能”，而是：

**把平台从“能跑通试点”推进到“能稳定驱动资料复审作业流”。**

这意味着近期最值得优先做的是：

1. 状态归一与单入口复审引导
2. 责任人与时效驱动
3. 项目 / 危大工程 / 分包对象建模

等这三层站稳后，再去系统性扩 AI 能力平台，会比现在边测边猜推进稳得多。
