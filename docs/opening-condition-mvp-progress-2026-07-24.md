# 开工条件核查最小 MVP 当前进度

更新时间：2026-07-24

## 当前结论

当前开工条件核查已经不是“从零开始”，而是处在 **最小 MVP 闭环验收前的收口阶段**。

已经具备的主链路：

```text
资料接入
  -> 正式资料核查
  -> 人工复核
  -> 报告生成
  -> DOCX 导出
  -> 归档
  -> 下一轮整改复审 run
```

但页面上仍存在一个认知问题：`依据与主数据/资产治理` 页面属于后续治理能力，不应作为判断最小 MVP 是否跑通的主入口。最小 MVP 验收应优先看资料接入、资料核查、人工复核、报告归档四个页面。

## 已完成

- 真实试点资料上传和初始化 run。
- 正式核查门禁：依据、主数据、知识库、资料包 readiness。
- checklist matrix：按核查项展示匹配、缺失、原因和人工复核关联。
- 人工复核队列：可确认、修正、延期、驳回。
- 报告资产：后端生成报告包，页面展示问题清单、交付状态、历史轮次和整改对照。
- DOCX 导出：通过 `HTTP_TOOLS_BASE_URL` 对接 http_tools 导出报告。
- 归档不可变：archived run 不允许继续正式核查、报告生成或重新初始化。
- 多轮次复审：归档后可发起新的 run，旧 run 作为历史记录保留。
- smoke 验收：后端 domain smoke、HTTP smoke、UI smoke 已形成基础护栏。

## 还缺的最小 MVP 收口项

1. 用真实样本稳定跑完至少两轮：
   - 第一轮：上传 -> 正式核查 -> 人工复核 -> 报告/DOCX -> 归档。
   - 第二轮：从报告页发起整改复审 -> 重新上传补件 -> 再次核查 -> 新报告 -> 新归档。
2. 记录每轮报告中的不符合项、整改项和历史变化。
3. 验证旧归档 run 只能查看，不能被误操作污染。
4. 保留一份真实样本验收记录，作为后续施工方案核查并行开发的参考。

## 当前不优先做

- 不继续深挖资产治理页的语义和交互。
- 不做完整多租户权限。
- 不做完整数据库迁移。
- 不大规模重构样式。
- 不为了某个识别误差无限调 provider。

## 页面操作建议

如果目标是验证最小 MVP，请按这个顺序操作：

1. `资料接入`：上传并初始化当前 run，确保 readiness 可以进入正式核查。
2. `资料核查`：查看 checklist matrix 是否产生匹配、缺失和待复核结果。
3. `人工复核`：关闭 open/deferred 项。
4. `报告归档`：生成报告、导出 DOCX、归档。
5. `报告归档`：从历史轮次或整改入口发起下一轮复审。

`资产治理（后续）` 页面只用于后续沉淀依据、主数据、知识库资产，不作为当前最小 MVP 是否跑通的判断入口。
