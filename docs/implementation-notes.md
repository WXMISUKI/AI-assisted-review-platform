# 施工方案审查交互 MVP 实现归档

## 本次交付

- 初始化 OpenSpec 与 CodeGraph。
- 创建 `interactive-review-mvp` OpenSpec 变更，包含 proposal、design、capability specs 和 tasks。
- 搭建 React + Vite + TypeScript 前端原型。
- 定义审查问题结构化模型：问题来源、状态、风险等级、文档锚点、问题说明、规范依据、建议修改、人工处理结果。
- 使用 mock 施工方案和 mock AI/人工问题实现工作台。
- 实现原文高亮、右侧问题卡片、状态筛选、双向定位、接受、拒绝、修改后接受、处理后方案预览。
- 增强 mock 工作台，支持在同一段落内拖选原文后新增人工标注。
- 人工标注复用 `ReviewIssue` 模型，创建后立即进入原文高亮、右侧卡片、筛选和处理后预览流程。

## 产品结论

施工方案审查平台的核心不是简单 AI 问题列表，而是以文档锚点为中心的协同审查工作台。前后端、AI、文档解析和报告生成都应围绕同一份 `ReviewIssue` 契约展开。

## 技术边界

本次原型使用 HTML 段落模拟文档，不处理 PDF 坐标、分页、缩放和跨页标注。后续接入真实 PDF 时，需要增加 PDF 标注 spike，优先验证以下能力：

- PDF 文字层选区是否能稳定映射到 `page + rects + text`。
- AI 返回的 `anchor.text` 是否能在解析文本中稳定回填。
- 缩放、滚动、跨页标注是否能保持定位准确。
- 已接受/已拒绝状态是否能同步到最终报告。

当前手动标注能力也只覆盖 HTML mock 场景：

- 支持同一段落内选区。
- 暂不支持跨段落选区。
- 暂不支持与已有标注重叠。
- 暂不持久化人工标注。
- 人工标注表单已改为选区附近的固定定位浮层，避免长文档中选区后跳到顶部填写。
- 当前浮层按选区矩形初始定位并做视口边界约束，暂不跟随滚动实时重定位；生产版可评估 floating-ui 或基于文档锚点重算位置。
- 新增“审查模式 / 审查修改模式”切换。默认审查模式只确认意见，不替换正文；审查修改模式下接受才会将当前建议文本应用到处理后全文预览。
- 接受/拒绝改为可逆选择，按钮高亮表示当前选择，不再锁定卡片，用户可随时切换。
- 人工标注支持删除确认。当前只允许删除 `source: manual` 的标注，AI 标注后续建议通过“作废/隐藏”而不是物理删除处理。
- 处理后预览改为连续文档流。长文档场景下比段落卡片更接近成熟文档审查工具；后续可扩展为“原文 / 修订后 / 红线对比”三视图。
