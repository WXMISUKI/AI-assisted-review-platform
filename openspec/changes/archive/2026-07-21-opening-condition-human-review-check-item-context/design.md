## Context

正式匹配会生成完整的 `checkItems`，其中已经有 `category`、`name`、`ruleExplanation`、`expectedEvidenceHints` 和 `evidenceIds` 的来源信息；但 `humanReviewQueue` 只保存 `targetId`，前端没有继续解析目标核查项。人工复核和报告台账因此发生了上下文丢失。

试点任务使用本地持久化适配器，任务数据可能来自当前版本或历史版本。方案必须兼容已有任务，且不得暴露原始 OCR 文本、私有对象 URL、provider secret 或无限长度文档内容。

## Goals / Non-Goals

**Goals:**

- 让每个复核项在页面和归档台账中可被监理人员独立理解。
- 让上下文成为任务快照的一部分，避免页面依赖不可重建的临时状态。
- 对旧任务提供有界、确定性的回填。
- 保持现有状态机、人工决策 API 和正式匹配行为不变。

**Non-Goals:**

- 不在本变更中建立项目/合同/危大工程的完整主数据模型。
- 不在本变更中实现知识库分段、人工确认入库或 provider 重新编排。
- 不重做 UI 视觉系统，不增加新的页面导航。
- 不把“资料存在”升级为“资料合规”，正式合规规则仍由后续任务组建设。

## Decisions

### 1. 在人工复核项上保存有界快照

新增可选字段 `targetLabel`、`category`、`subCategory`、`ruleExplanation`、`expectedEvidenceHints`。正式匹配创建队列时直接写入；归一化时限制长度和数组数量。

选择任务快照而不是只在前端查找的原因是：任务可能被归档、checklist definition 可能在后续版本变化，历史复核和报告必须保持当时语义。

### 2. 保留 `targetId` 作为稳定主键

`targetId` 继续作为核查项关联键，新增字段只负责显示和历史可读性。这样不改变现有决策接口，也不影响旧数据和其他 `basis/master_data/report` 类型的复核项。

### 3. 旧任务采用后端归一化回填

读取任务时，如果队列项缺少快照字段，则按 `targetType=check_item` 查找同任务 `checkItems`，再回退到 `checklistDefinition`。回填结果写入归一化任务返回值，持久化迁移保持轻量，不执行全量数据库迁移。

### 4. 页面展示证据引用而非原文

复核卡片展示证据 ID、文件名/定位信息和规则提示，不直接展示未裁剪 OCR 原文。用户若需要后续查看原文，应由独立证据查看能力承接。

## Risks / Trade-offs

- [历史任务字段缺失] -> 读取时按目标 ID 回填，无法回填时保留编号并显示“未找到核查项快照”。
- [队列快照与 checkItem 后续修改不一致] -> 以人工复核项快照为历史事实，当前 checkItem 仅作为缺失字段回退。
- [页面信息密度上升] -> 采用分类、名称、编号、原因、证据的固定层级，不引入额外复杂交互。
- [错误字段过长] -> 后端归一化统一限制字符串和提示数组长度。

## Migration Plan

1. 发布后端归一化和正式匹配生成逻辑。
2. 发布前端类型和人工复核/报告台账展示。
3. 用现有真实试点任务刷新人工复核页，确认三项待复核内容均能看到分类、名称和证据上下文。
4. 若历史任务无法回填，仅提示缺少快照，不阻断任务决策。

## Open Questions

- 项目、合同包、专业/危大工程和知识库的正式实体模型另立变更，在真实试点完成后根据实际资料边界确认。
