## Context

项目当前是全新仓库，仅包含 README。目标产品是面向监理方的施工方案 AI 审查平台：施工方提交方案，监理通过平台查看 AI 识别的问题，并在原文位置完成复核、采纳、拒绝或手动修订。

前期调研表明，完整产品不应只是一个后端 AI 工作流，而应是“文档锚点批注 + 侧边问题闭环 + 结构化 AI 输出 + 后续报告归档”的协同平台。本次设计只覆盖可验证交互 MVP，用 mock 文档和 mock AI 问题验证产品定位与核心状态流。

## Goals / Non-Goals

**Goals:**

- 建立 React/Vite 前端原型，用户一条命令即可运行。
- 用 mock 数据呈现施工方案正文、AI 高亮问题、右侧审查卡片和处理统计。
- 支持点击卡片定位原文、点击原文标注聚焦卡片。
- 支持接受 AI 建议、拒绝问题、编辑建议后接受。
- 显示处理后方案预览，帮助验证“最终比对后的方案”产品方向。
- 用 TypeScript 明确审查问题、锚点、状态和处理结果的数据结构。

**Non-Goals:**

- 不接入真实 LLM、Dify、LangGraph 或规范知识库。
- 不解析真实 PDF/DOCX，不实现 PDF 坐标级标注。
- 不实现登录、权限、数据库、文件上传或服务端 API。
- 不生成正式 PDF/Word/Excel 报告。
- 不把原型视为最终生产 UI；本阶段代码以验证交互为主。

## Decisions

### Decision 1: 先做 HTML 段落原型，而不是直接集成 PDF 标注

使用结构化段落渲染施工方案正文，每个问题通过 `paragraphId + startOffset + endOffset` 锚定。这样能最快验证“原文标注与卡片联动”的体验，并提前沉淀后续 PDF/Word 解析也能复用的 anchor 契约。

备选方案是直接接入 `react-pdf-highlighter`。它更贴近最终 PDF 场景，但会把首轮验证复杂度转移到 PDF 解析、坐标归一化和样式适配上，不利于先确认产品交互。

### Decision 2: 问题状态由前端内存管理

本阶段使用 React state 管理问题状态，避免引入数据库和后端 API。所有处理动作都会实时更新问题卡片、原文高亮和处理后方案预览。

后续接入后端时，可以把相同状态迁移为持久化事件：`accepted`、`rejected`、`modified`。

### Decision 3: 以 issue model 作为前后端、AI 与文档解析之间的统一契约

审查问题模型必须包含锚点、原因、依据、建议、状态和处理结果。AI 不应只返回自然语言列表，而应返回可被 UI 直接消费的结构化对象。

后续真实链路可以由 MinerU/PDF 解析生成段落结构，由 Dify/LangGraph 生成问题 JSON，由前端渲染高亮并收集人工处理结果。

### Decision 4: 采用企业工具型界面，不做营销页

首屏直接进入审查工作台。布局采用顶部项目摘要、左侧文档正文、右侧问题面板、底部/侧区预览的密集工具界面。视觉上强调可扫描、可定位、可处理，而不是品牌展示。

## Risks / Trade-offs

- HTML 段落原型不能暴露 PDF 坐标偏移问题 -> 后续阶段必须新增 PDF 标注 spike，验证缩放、跨页、高亮恢复。
- Mock AI 数据可能低估真实模型锚点不稳定问题 -> 后续 AI 输出必须做 anchor_text 精确匹配、模糊回退和人工确认。
- 纯前端状态无法验证多人协作与审计 -> 本阶段只验证单人交互，企业级审计在后端设计阶段补齐。
- 原型代码可能被误当生产代码长期保留 -> 文件和文档中明确标记 MVP/prototype，后续应吸收为正式架构或删除重写。

## Migration Plan

1. 创建前端 MVP 应用和 OpenSpec 文档。
2. 使用 mock 数据验证核心交互。
3. 通过简单类型检查确认代码可运行。
4. 将本次 change 归档，并同步主规格。
5. 下一阶段在新 change 中接入真实 PDF 标注和解析能力。

## Open Questions

- 第一阶段真实文档优先支持 PDF 还是 DOCX？
- 后续 AI 编排优先使用 Dify 作为低代码引擎，还是 LangGraph 作为代码级状态机？
- 规范知识库来源、授权和版本管理方式如何确定？
- “接受”在业务上是表示认可 AI 问题成立，还是表示采纳 AI 修改并更新方案正文？本 MVP 按“采纳建议并更新处理后方案”实现。
