## Context

当前项目已具备单项目试点的最小闭环、可重复 run、归档只读约束、历史轮次对照和报告页基础摘要。但从真实工程监理交付的角度看，平台仍缺少两层关键沉淀：

1. 核查项在失败、阻塞、待人工复核时，还没有稳定映射到“问题类型 / 法规依据 / 整改要求 / 复核建议”这一层问题资产。
2. 报告生成仍偏向平台内部统计与运行诊断，尚未形成可复用的“问题交付包”结构，无法稳定支撑报告页重构、后续 docx 回填或问题型智能体复用。

我们已经确认下一阶段不优先做完整多租户、数据库迁移或 UI 美化，而是继续把“真实资料可重复运行的试点交付包”做实。因此本设计聚焦平台 own 的问题资产和报告交付结构。

## Goals / Non-Goals

**Goals:**

- 为开工条件试点建立稳定的问题分类模型，至少覆盖问题类型标识、显示名称、风险等级、法规依据、整改要求和复核建议。
- 扩展试点 `checkItem`、人工复核和 `report package diagnostics`，让报告生成能够产出平台自有的结构化问题交付包。
- 保持向后兼容：已有试点任务、已有报告摘要、已有人工复核链路继续可用。
- 为后续 docx/html 报告生成、原表回填、法规整改智能体、问题复核智能体保留可扩展字段。

**Non-Goals:**

- 本次不直接实现 `docxToHtml` / `htmlToDocx` 导出与原表回填。
- 本次不引入新的数据库或远端持久化方案，继续沿用当前本地 store 的领域扩展。
- 本次不做完整 UI 视觉重构，只补足首批可消费的数据结构与必要的报告展示语义。
- 本次不一次性实现十二类问题的完整智能识别引擎，只先固化规则资产承载结构与当前试点映射。

## Decisions

### 1. 采用“问题分类注册表 + check item 派生字段”而不是只在报告阶段临时拼装

决定：

- 在领域模型中新增开工条件问题分类定义与结构化字段。
- `checkItem` 在匹配完成后即可派生出 `issueTypeId`、`issueTypeLabel`、`riskLevel`、`legalBasis`、`rectificationRequirement` 等字段。

原因：

- 这样问题资产从核查阶段开始就成为平台事实，而不是报告页特有逻辑。
- 后续人工复核、整改闭环、报告导出和智能体复用都能消费同一套字段。

备选方案：

- 只在报告生成时根据名称和 verdict 临时推断问题类型。

未选原因：

- 这会让问题语义散落在报告页或 store 的一次性逻辑里，后续难以复用，也不利于状态与资产统一治理。

### 2. 报告交付包采用“结构化 findings 列表 + 摘要诊断并存”的扩展模式

决定：

- 保留现有 `packageDiagnostics` 摘要字段。
- 新增结构化 `findings`、`summaryByIssueType`、`nextRectificationAdvice` 等交付字段。

原因：

- 现有摘要已经支撑了页面和 smoke，不必推翻。
- 增量扩展可以兼容历史数据，同时给报告页更强语义。

备选方案：

- 直接用新的报告对象完全替换旧 `packageDiagnostics`。

未选原因：

- 会提高迁移风险，也会让现有试点和测试一次性破坏过大。

### 3. 问题分类先以内置注册表实现，并为未来智能资产绑定预留挂点

决定：

- 首批问题分类规则以内置 registry 实现，映射当前开工条件核查项目。
- 分类定义中预留 `agentAssetId`、`promptAssetId`、`templateId` 等可选挂点，但本轮不要求这些资产已有 CRUD。

原因：

- 当前最值钱的是先把平台事实统一下来。
- 内置 registry 足够支撑首轮试点，又不会堵死后续接资产平台。

备选方案：

- 直接先做问题分类配置后台或 agent 资产管理联动。

未选原因：

- 超出本轮范围，会把投产前最核心的报告交付包问题拖慢。

### 4. 对历史 task 和报告采用“缺省回填”兼容策略

决定：

- 领域 normalize 层和 store 派生逻辑都允许缺失新字段。
- 报告生成时如果旧 task 未显式存储问题分类字段，则通过 registry 进行派生补齐。

原因：

- 当前本地试点数据已存在多轮历史 run，不能因为这次结构升级导致旧数据不可读。

### 5. 将“原表回填/导出”明确放在后续切片，通过 report package contract 衔接

决定：

- 本设计只要求报告资产产出稳定交付包事实。
- 后续文档导出器只读取这一层事实，不直接依赖页面拼装或 provider 原始输出。

原因：

- 这能让后续接你提供的 `docxToHtml`、`htmlToDocx` 接口时更稳，避免再次返工领域模型。

## Risks / Trade-offs

- [风险] 当前问题类型与法规依据首批映射仍然偏规则化，可能无法完全覆盖真实项目的所有变体。  
  → Mitigation：以 registry + 可选覆盖字段实现，后续可逐步引入可配置资产或子智能体。

- [风险] 扩展 `report package diagnostics` 会让 store 结构更复杂。  
  → Mitigation：保持旧字段不变，仅新增结构化字段，并在 normalize 层集中兼容。

- [风险] 如果前端立刻全面消费新字段，可能和旧 run 数据显示不一致。  
  → Mitigation：前端采用“优先消费新字段，缺失则回退旧字段”的渐进策略。

- [风险] 本次不直接做 docx 导出，用户可能仍觉得报告交付件还不够“最终形态”。  
  → Mitigation：在 tasks 和规格中明确把导出挂点保留下来，下一切片优先接文档交付。
