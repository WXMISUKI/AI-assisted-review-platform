## Why

当前开工条件试点已经能跑通“资料接入 -> 正式核查 -> 人工复核 -> 报告生成 -> 归档”的最小闭环，但距离真实试点可交付仍差一层关键能力：平台还没有把核查问题沉淀为稳定的问题分类资产，也没有把报告输出提升为接近监理交付件的结构化报告包。

这一步现在最值得做，因为用户已经用 Dify 工作流验证过“核查项识别 + 问题分类 + 法规依据 + 整改要求 + 最终报告”的业务价值。平台需要把这部分能力从临时工作流表达，提升为平台 own 的领域模型、API 契约和报告语义，为后续子智能体复用、法规整改智能体和原表回填接口留出稳定挂点。

## What Changes

- 新增开工条件试点的问题分类能力，统一表达问题类型、风险等级、法规依据、整改要求、复核建议和后续智能资产挂点。
- 扩展试点任务 `checkItem` 与报告诊断数据结构，使其能够输出面向交付的结构化问题清单，而不只是内部统计摘要。
- 扩展报告交付语义，使报告页能够按问题分类、风险等级、法规依据、整改要求和人工结论生成可读的交付包。
- 扩展试点运行 API 合同，使报告响应能够稳定返回结构化问题交付包，供前端、后续导出器和智能体资产复用。
- 为后续 `docxToHtml` / `htmlToDocx` 原表回填与导出保留扩展字段和交付包语义，但本变更不直接实现文档导出。

## Capabilities

### New Capabilities
- `opening-condition-issue-taxonomy`: 定义开工条件核查问题分类、法规依据、整改要求、交付语义以及未来智能资产复用挂点。

### Modified Capabilities
- `opening-condition-report-findings-delivery`: 报告交付从统计摘要提升为结构化问题交付包，要求展示问题分类、法规依据、整改要求、人工结论和历史交付语义。
- `opening-condition-pilot-operational-api`: 试点 API 需要返回结构化问题交付包和问题分类字段，供报告页、后续导出器和智能资产使用。

## Impact

- 受影响代码：
  - `src/domain/openingConditionPilot.ts`
  - `server/openingConditionPilotStore.mjs`
  - 报告页与相关前端 view model / rendering 模块
- 受影响系统：
  - 开工条件试点本地后端状态存储
  - 报告页交付语义
  - 后续报告导出器、子智能体 / 提示词资产、法规整改资产的接入边界
- 不引入新的外部依赖；本次以现有本地试点数据结构扩展为主。
