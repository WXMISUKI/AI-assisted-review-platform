## Context

项目已经完成：

- 单项目试点闭环
- 问题分类资产与结构化报告交付包
- 历史轮次、归档只读、整改复审语义

但接下来如果要继续靠近真实交付，还缺两个平台级层面：

1. 智能资产并未细化为开工条件专属的可复用目录。当前 `agentAssets.ts` 里只有一个宽泛的“开工条件核查智能体”，还不足以承接“十二类问题审查”“法规整改”“报告回填/导出”这些后续能力。
2. 报告交付包虽已具备 findings 语义，但还没有 exporter-ready handoff。后续无论是外部 `docxToHtml/htmlToDocx` 服务、模板回填器，还是 Dify 的文档回填工作流，都缺少稳定输入 contract。

这次设计的目标，是先把这些能力沉淀为平台可绑定的资产与 contract，而不急着直接发起真实外部接口调用。

## Goals / Non-Goals

**Goals:**

- 将开工条件专属智能资产细分为可展示、可绑定的资产条目，包括问题类型审查、法规整改、报告回填/导出适配器。
- 为提示词、模板、输出 schema 与问题类型资产建立稳定绑定关系。
- 扩展报告交付包，增加 exporter-ready handoff 摘要、模板绑定、导出适配状态和输入清单。
- 在平台资产页和报告页上露出这些结构，便于后续接真实外部接口。

**Non-Goals:**

- 本次不直接接入真实 `docxToHtml` / `htmlToDocx` 服务。
- 本次不实现真正的多 agent 执行编排或异步任务调度。
- 本次不做完整资产 CRUD 后台，只先补目录、结构和绑定关系。

## Decisions

### 1. 采用“资产目录先行，真实执行后接”的策略

决定：

- 先把开工条件智能资产目录做细，包括：
  - 问题类型审查智能体
  - 法规整改智能体
  - 报告回填/导出适配器
- 真实接口调用和执行编排放到后续切片。

原因：

- 当前最大风险不是没有调用某个接口，而是平台没有稳定资产边界。
- 目录先行后，后续每个真实能力都能挂到已有资产项上。

### 2. 原表回填使用“export handoff contract”作为唯一输入边界

决定：

- 在报告资产 `packageDiagnostics` 中新增 `exportHandoff` 结构，描述：
  - 交付类型
  - 模板/适配器标识
  - 输入对象清单
  - findings 计数
  - 当前适配状态
  - 下一步动作

原因：

- 后续真实导出器、Dify 回填节点或平台内 worker 只读这一层，不直接依赖页面拼装。

### 3. 问题类型与智能资产通过稳定 id 绑定，而不是用文案硬编码

决定：

- 开工条件问题类型与智能资产之间使用 `issueTypeId -> agentAssetId/promptAssetId/templateId` 绑定。
- 报告导出适配器单独使用 `exportAdapterId` / `templateId` 表达。

原因：

- 后续资产版本化、替换提示词、增加法规整改 agent 时不需要反复改报告逻辑。

## Risks / Trade-offs

- [风险] 先做资产目录会让用户短期看到“待接入”的适配器。  
  → Mitigation：明确标识这是平台预留挂点，重点是避免后续接接口时返工。

- [风险] export handoff 字段过早设计可能和真实接口略有偏差。  
  → Mitigation：只定义平台稳定必需字段，外部接口参数映射留给后续 adapter 层。

- [风险] agent 资产目录继续扩张后，页面可能显得拥挤。  
  → Mitigation：这轮先做结构和条目，不做大面积视觉重构，后续统一整理资产页布局。
