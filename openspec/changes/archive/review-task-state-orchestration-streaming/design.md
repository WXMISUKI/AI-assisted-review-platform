## Context

当前项目已经具备文档库、OCR 提交、审查流式 loading 页、审查工作台和结果页，但任务状态推进分散在 `App.tsx`、`reviewSessionService.ts`、`backendConnectivity.ts` 和页面组件之间。虽然能跑通 mock 流程，但状态恢复、流式重放、后端替换和详情页锁定逻辑仍然依赖页面编排，容易在后续接入真实 OCR、LLM 和 SSE 时失去一致性。

本次设计的目标，是把“文档审查任务如何从上传走到完成”收束成一个明确的领域编排层，让文档库、loading 页和审查工作台都消费同一份任务生命周期结果。

## Goals / Non-Goals

**Goals:**

- 建立统一的任务状态编排契约，覆盖上传、OCR、结构恢复、审查准备、就绪、完成和失败。
- 让 OCR 进度、结构恢复进度和审查流式阶段都能从同一个任务快照中恢复。
- 让 loading 页可以在任务未就绪时保持锁定，并展示真实可读的阶段信息。
- 让现有 mock 流程与未来后端异步任务、SSE 事件能够共享同一套字段。
- 减少页面组件里的业务状态拼接，把复杂逻辑下沉到 `src/domain`。

**Non-Goals:**

- 不在这次变更中接入真实后端 agent 编排。
- 不在这次变更中替换现有 mock 文档数据源。
- 不在这次变更中改造结果页导出能力。
- 不在这次变更中引入新的数据库或消息队列依赖。

## Decisions

### 1. 用领域编排层承载状态推进，而不是让页面自己算状态

**Decision:** 新增一个领域层编排模块，负责把上传、OCR、结构恢复和审查流式进度整合为统一的 task lifecycle snapshot。

**Why:** 现在的 `App.tsx` 已经承担了太多责任，页面既在做导航，又在推进 OCR 轮询，又在更新流式阶段。把这些逻辑留在页面里，会让后续的后端替换和恢复逻辑越来越难维护。

**Alternatives considered:**
- 继续放在 `App.tsx`：改动最少，但会继续膨胀，且状态恢复难以统一。
- 只拆 service，不建编排层：能稍微减负，但 OCR、结构恢复和 streaming 之间仍然没有一个明确的总入口。

### 2. 任务快照以“可恢复读模型”为主，而不是把后台实现细节暴露给页面

**Decision:** 任务快照保留状态、阶段、段落/章节上下文、OCR 进度和流式阶段索引，但页面只消费一个统一摘要和少量结构化字段。

**Why:** 页面不应该知道 OCR 怎么轮询、流式事件怎么拼、后端 future agent 如何拆分阶段。它只需要知道“现在任务卡在哪一步”和“下一步是否可进入详情”。

**Alternatives considered:**
- 直接暴露后端原始事件：灵活，但耦合太强，前端会被实现细节绑死。
- 只存最终结果：简单，但失去 loading 页和恢复能力。

### 3. 统一流式事件契约，兼容 mock 和 SSE

**Decision:** 复用当前 review stream 的阶段字段，但要求事件同时携带 stage、paragraph、agent 和 issue summary 等信息，能直接映射回任务快照。

**Why:** 这样 mock 流和真实 SSE 流可以共用同一条 UI 通道，未来替换只改数据源，不改页面结构。

**Alternatives considered:**
- 只保留简单进度百分比：容易实现，但无法展示“当前在处理哪个段落”。
- 依赖双向通信：复杂度高，不适合当前阶段。

### 4. loading 页保持“锁定但可观察”

**Decision:** 任务未就绪前，详情页继续作为锁定态观察窗口，只读展示 OCR 或审查准备过程，直到状态进入 ready 再开放交互。

**Why:** 这符合企业级审查平台的习惯：用户可以看到后台在处理什么，但不能提前改结果或误操作。

**Alternatives considered:**
- 直接跳过 loading 进入可编辑页面：体验更快，但容易造成状态错乱。
- 完全隐藏处理中页面：丢失可信感，也不利于排查任务卡点。

## Risks / Trade-offs

- [Risk] 状态快照字段太多，容易再次膨胀。→ [Mitigation] 只保留页面真的需要的恢复字段，其余后台细节放在 service 层。
- [Risk] mock 和未来后端在阶段命名上不一致。→ [Mitigation] 在编排层增加规范化映射，前端只看统一枚举。
- [Risk] loading 页锁定时间长，用户会以为卡死。→ [Mitigation] 强化阶段文案、百分比和当前段落/章节显示。
- [Risk] 继续推进时 App.tsx 仍然偏大。→ [Mitigation] 这次只抽状态编排，不顺手把所有 UI 再重构一遍。

## Migration Plan

1. 先在 `src/domain` 新增任务状态编排模块和统一快照/事件类型。
2. 将 `reviewSessionService`、OCR 轮询和流式 stage 推进接入新编排层。
3. 让文档库和 loading 页优先消费编排层输出，再逐步收窄页面内状态拼装。
4. 跑 `typecheck` 和关键页面 smoke 验证，确认状态流转不回退。
5. 后续真实后端接入时，只替换事件来源与 OCR/agent 调用，不改页面契约。

## Open Questions

- 未来后端 SSE 的事件命名是否要与当前 mock 事件完全一致，还是允许服务层做规范化映射？
- OCR 完成后进入审查准备阶段时，是否需要额外拆出“结构恢复中”的独立状态展示？
- loading 页是否需要支持用户中断后自动恢复到当前阶段，而不仅仅是刷新后恢复？
