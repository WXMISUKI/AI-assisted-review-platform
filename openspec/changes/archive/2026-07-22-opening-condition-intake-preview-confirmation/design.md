## Context

现在的资料接入页已经有：

1. 真实试点上传入口
2. 试点执行台
3. Trial Intake Overview
4. 试点诊断摘要

但它仍然缺一个关键层次：**上传后系统识别成了什么，正式确认前先让人看一遍**。当前 basis/master-data 的识别结果更多是在“依据与主数据”页的发布治理面中回看，而不是在接入入口附近就解释清楚。

对真实试点来说，这会带来两个问题：
- 操作者上传后不容易立即理解“系统准备入库什么”。
- “确认主数据 / 发布依据”这些动作看起来像流程门禁按钮，而不是“正式确认识别结果”的业务动作。

## Goals / Non-Goals

**Goals:**
- 在资料接入页增加一块识别预览确认工作台。
- 让 basis 和 master-data 候选记录用 operator-facing 方式展示，强调“正式入库前预览”。
- 让现有的发布依据、确认主数据动作在预览语义中变得更可理解。
- 与已有发布治理面保持口径连续，但不重复做整页目录视图。

**Non-Goals:**
- 不做逐条字段编辑器。
- 不新增新的后端解析或保存 API。
- 不在本轮引入正式审批流、责任人指派或多级审核。
- 不改报告页或历史复审页。

## Decisions

### 1. 在资料接入页增加“识别预览确认工作台”，不另开新页面

这层离上传入口越近越有价值。操作者上传后能立刻看到 basis/master-data 候选，再决定是否正式确认，而不需要跳页去理解。

### 2. 预览视图聚焦“当前 run 候选”，不替代发布治理目录

资料接入页只关心当前 run 及其候选：
- 当前 basis 候选
- 当前 run 所需主数据候选
- 当前仍待处理的低置信/待确认项

发布治理页继续负责全局目录与异常历史。两页分工不同：
- 接入页：看“这次上传识别了什么”
- 治理页：看“工作区资产目录现在是什么状态”

### 3. 重用现有 basis/master-data 状态，不新增新枚举

本轮直接基于已有状态做语义映射：
- basis: `pending_confirmation / confirmed / published`
- master data: `provisional / confirmed / human_approved / published`

通过前端视图把这些状态翻译成“待确认”“待发布”“已可用于当前 run”等 operator-facing 文案。

### 4. 预览区直接挂接现有批量动作

目前系统已有：
- 发布当前 run 依据
- 确认当前 run 主数据

本轮不新造动作，只在预览工作台中把这些动作解释成“正式纳入平台资产”的确认步骤。

## Risks / Trade-offs

- [接入页信息继续增多] -> 用独立工作台分区承载，不把信息塞进现有 Trial Intake Overview 卡片里。
- [预览页和治理页可能重复] -> 明确一个看当前 run 候选，一个看工作区目录。
- [缺少逐条编辑能力] -> 本轮先解决“看清楚再确认”，逐条修订留到后续更大的入库治理阶段。
