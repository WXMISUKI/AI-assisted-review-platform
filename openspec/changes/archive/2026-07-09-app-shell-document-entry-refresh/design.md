## Context

当前平台已经具备登录、文档库、知识库、数据资产与审查工作台的基础形态，但壳层层级偏暗、用户区可见性不足，文档详情的承接方式仍偏“嵌套子页”，上传入口和预览入口也还不够统一。这个变更的目标是先把平台的入口骨架校准，让后续审查交互落在更成熟的页面结构上。

## Goals / Non-Goals

**Goals:**
- 让登录后的全局壳层更像企业级审查平台，而不是临时 demo 容器。
- 让用户信息、角色与退出操作始终可见且位置稳定。
- 让文档库入口支持拖拽与点击上传，并保留可选元信息。
- 让文档详情成为独立工作页，增强页面切换的确定感。
- 为主题切换预留统一 token 入口，避免后续重复改造样式。

**Non-Goals:**
- 不在本变更中实现真实路由系统。
- 不在本变更中实现真实文件解析、后端任务、SSE 或 WebSocket。
- 不在本变更中重做审查工作台的标注算法或划词浮层。
- 不在本变更中接入真实知识库或数据资产后端。

## Decisions

### 1. 全局壳层保持可见，但详情页脱离壳层主内容区
文档库、知识库和数据资产继续使用同一个平台壳层，以维持导航一致性；但当用户进入文档详情时，页面应切换到独立的全屏工作页，并在左上角提供返回按钮。这样做比在右侧主体中继续嵌套一层更符合审查工作台的注意力模型，也更方便后续改为真正的路由页。

**备选方案考虑过：**
- 继续在壳层右侧承载详情页。实现最省事，但视觉层级不对，容易像“页面里的页面”。
- 把壳层完全拆成两个独立应用壳。视觉更干净，但代价更高，不适合当前阶段。

### 2. 主题切换采用 CSS 变量驱动，而不是组件内分支改样式
浅色主题作为默认，深色主题作为可切换选项，统一通过 CSS 自定义属性承载颜色、边框、背景和文本层级。这样可减少组件内条件判断，也便于后续将登录页、文档库和详情页统一在同一套设计 token 下。

**备选方案考虑过：**
- 组件内分别写两套 class。短期快，但后续维护成本高。
- 直接只做单一浅色主题。最稳，但错过用户需要的深浅切换能力。

### 3. 上传入口以“拖拽 + 点击选择”为统一交互
上传区域应同时支持拖拽和文件选择按钮，并允许文件名、项目名称作为可选字段。默认使用文件名和“未知项目”，这样既不强迫用户补全信息，也给后续结构化录入留出空间。

**备选方案考虑过：**
- 强制填写全部元信息。完整，但会抬高上传门槛。
- 只保留拖拽。过于单薄，不符合成熟平台预期。

### 4. 快捷预览从文档库主路径中移除
文档库的首要任务是“找到文档、开始任务、进入详情”，不是在列表页里做过多预览消费。预览类交互保留到文档详情内部或后续独立报表页中更合适。

## Risks / Trade-offs

- [风险] 详情页改为独立全屏后，短期内会让页面状态切换逻辑更明显。→ [缓解] 先保持本地状态驱动，后续再平滑迁移到路由。
- [风险] 浅色化侧栏后，部分信息密度可能显得不够强。→ [缓解] 通过分组、边框、字号和状态标签补回层次，而不是继续加深背景。
- [风险] 主题切换若做得过早，可能诱发样式分叉。→ [缓解] 只做 token 层，不在组件内复制两套布局。

## Migration Plan

1. 先调整壳层和文档库的页面结构，让入口关系清晰。
2. 再把上传与文档进入改成可点击、可拖拽、可恢复的状态流。
3. 最后统一补齐样式 token 与主题切换。
4. 如果后续接入真实路由，只需把当前页面状态映射到对应路径即可。

## Open Questions

- 详情页未来是否要由真实路由承载，还是保留单页状态切换再统一迁移？
- 主题切换是否需要记忆到本地存储，还是先默认跟随系统？
