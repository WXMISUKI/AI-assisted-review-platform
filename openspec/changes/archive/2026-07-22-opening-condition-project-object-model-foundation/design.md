## Context

当前开工条件门户已经具备单工作区闭环能力，但 `OpeningConditionWorkspace` 仍然是扁平对象，页面展示也把“工作区”直接等同于业务上下文。这会带来两个问题：

1. 无法清晰表达“同一项目下多个危大工程/审查对象”的边界。
2. 后续的 run 历史、知识库、依据发布和责任流只能挂在平面 workspace 名称上，难以扩展。

这轮目标不是直接引入数据库级对象模型，而是先在前端与样本域模型中引入最小分层语义，并让现有试点链路都消费这层语义。

## Goals / Non-Goals

**Goals:**
- 为开工条件门户建立“项目 -> 审查对象 -> 参建主体”的最小对象模型。
- 让工作区切换从平面列表升级为按对象分组的上下文切换。
- 让当前 run、历史轮次、知识库和资料接入动作都显式绑定到对象上下文。
- 保持与现有试点链路兼容，不打断已有正式核查、人工复核和报告归档流程。

**Non-Goals:**
- 不新增后端持久化表或 API 字段。
- 不做多租户权限、组织架构或人员级待办分派。
- 不在本轮实现依据入库预览与发布治理。
- 不把“现场核查对象”扩展成独立移动端或巡检流程。

## Decisions

### 1. 保留 workspace 作为技术容器，引入对象层元数据

不直接移除 `workspace` 概念，而是在 `OpeningConditionWorkspace` 上补充：

- `projectId` / `projectCode`
- `reviewObjectId` / `reviewObjectName` / `reviewObjectType`
- `participantEntityId` / `participantEntityName`

原因：
- 现有页面、run id 和本地样本都已经以 workspace 为主键。
- 先扩展元数据比重写主键关系更稳。
- 后续如果要下沉到后端，可以把这层字段直接迁移为正式对象引用。

备选方案：
- 直接新增独立 `Project / ReviewObject / ParticipantEntity` 持久化模型。
  - 未选，因为本轮重点是试点产品骨架，不是后端建模工程。

### 2. 通过派生 catalog 组织页面，而不是在页面里手工 groupBy

在领域层新增 catalog 派生函数，把扁平 workspace 列表转换为：

- 项目摘要
- 审查对象摘要
- 主体摘要

页面只消费 catalog 结果，不在组件内再次拼装层级。

原因：
- 避免页面继续堆积业务规则。
- 后续加入更多对象字段时，只需改领域派生函数。

### 3. 当前 run 与历史 run 继续按 workspace 取数，但 UI 显示对象边界

由于后端当前仍按 workspace 维度隔离任务，本轮继续沿用 `workspaceId -> tasks` 的取数方式；但所有 operator-facing 展示都改成对象语义。

原因：
- 能最快把“试点可用性”往前推进。
- 不引入后端兼容风险。

### 4. 工作区概览页升级为“对象选择台”而不是“平面工作区列表”

概览页将展示：

- 当前项目摘要
- 当前审查对象摘要
- 当前参建主体摘要
- 同项目下可切换对象/主体列表

原因：
- 这比继续平铺 workspace 卡片更符合真实监理作业方式。
- 也为后续“历史 run / 依据版本 / 知识库版本”挂到对象上提供入口。

## Risks / Trade-offs

- [对象层仍是前端样本模型，不是后端正式模型] → 通过稳定字段命名和 catalog 派生函数，降低未来下沉成本。
- [当前仍以 workspaceId 作为 run 隔离主键] → 本轮明确把 workspace 视为技术容器，对外展示对象语义，后续再替换后端主键关系。
- [页面信息密度会上升] → 通过摘要卡片和分层列表收敛展示，不恢复营销式大卡片布局。
- [现有样本数据较少，层级切换效果可能有限] → 补充 2-3 组同项目不同对象/主体的样本，保证结构真实可见。
