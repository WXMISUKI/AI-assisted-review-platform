## Why

当前开工条件试点虽然已经能跑通单条 run，但“工作区”仍然是扁平列表，无法表达一个监理同时管理多个项目、同一项目下多个危大工程/审查对象、以及不同分包或参建主体的真实作业结构。现在继续打磨局部页面价值不高，最该补的是对象层骨架，这样后续的复审历史、知识库绑定、依据发布和责任流才能真正挂在正确的业务对象上。

## What Changes

- 为开工条件工作区引入最小对象模型：项目、审查对象、参建主体三层上下文。
- 将当前扁平工作区数据改造成可分组、可切换、可展示对象边界的工作台视图。
- 让当前 run、历史轮次、知识库和资料接入动作都显式绑定到选中的对象上下文，而不是只显示一个平面 workspace 名称。
- 在工作区概览中补充对象层摘要，帮助操作者快速看清“当前在哪个项目、哪个审查对象、哪个主体”。
- 保持本轮为前端与本地样本层的基础改造，不引入多租户权限、数据库迁移或后端持久化新模型。

## Capabilities

### New Capabilities
- `opening-condition-project-object-model`: 定义项目、审查对象、参建主体及其与 trial run、知识库、历史轮次的最小绑定关系。

### Modified Capabilities
- `opening-condition-context`: 工作区上下文从平面项目/合同段/参建单位，扩展为项目/审查对象/参建主体分层选择与隔离。

## Impact

- `src/domain/openingConditionReview.ts`：补充对象模型类型、样本数据和上下文派生工具。
- `src/App.tsx`：按对象上下文驱动工作区选择、任务切换和 run 初始化入参。
- `src/productWorkspacePages.tsx`：将工作区概览与切换入口升级为分层对象视图。
- `src/styles/opening-condition.css`：补充对象摘要、对象切换列表和上下文信息样式。
