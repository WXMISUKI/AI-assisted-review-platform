## Context

当前平台已经完成单项目真实试点闭环：上传资料、正式核查、人工复核、报告归档、历史轮次查看和归档后发起下一轮复审。下一阶段最能推进投产的能力，是把多轮复审之间的问题变化讲清楚，让监理能判断补件是否有效。

现有前端已经能拿到 workspace 内多个 `OpeningConditionPilotTask`，每个 task 包含 `checkItems`、`humanReviewQueue`、`reportAsset` 和时间信息。因此本轮优先在前端构建相邻 run 的差异视图，不引入新的后端存储或复杂数据模型。

## Goals / Non-Goals

**Goals:**
- 在报告页展示上一轮与当前轮的整改闭环对照。
- 将问题按“已整改、仍未整改、本轮新增、待人工判断”归类。
- 让历史详情页在选择某一轮时，也能显示该轮相对于上一归档轮次的差异。
- 保持历史 run 只读，不修改归档任务。

**Non-Goals:**
- 不做正式整改通知单、docx/pdf 导出。
- 不做跨工作区、跨危大工程的全局统计。
- 不扩展 RBAC、多租户或数据库迁移。
- 不优化底层 OCR/LLM 匹配准确率。

## Decisions

### 1. 用 check item id 作为本轮差异主键

当前 checklist 已经将核查项标准化成 `oc-check-*` 或 `item-*` 级别标识。用 check item id 对齐相邻 run，能最小成本判断问题是否延续。

备选方案是用核查项名称模糊匹配，但名称可能被人工调整，短期更容易误判。

### 2. 差异状态由前端派生

本轮先用已有任务数据派生闭环状态，不写入后端。原因是试点阶段最需要验证报告是否有用，而不是先固化数据库结构。

后续若报告导出或审计要求增强，可把这些派生结果写入 reportAsset 的结构化字段。

### 3. 只对问题类 item 做闭环归类

对比范围聚焦 failed、blocked、needs_human_review、warning。已通过项只在判断“上一轮问题是否已整改”时作为目标状态使用。

这避免把完整核查矩阵变成庞大对照表，保持报告页可扫读。

## Risks / Trade-offs

- [核查项 id 不稳定导致对比失真] → 保留标题和分类展示，后续可增加名称兜底匹配。
- [前端派生结果不具备正式审计持久性] → 当前定位为试点交付视图，后续报告资产结构稳定后再后端持久化。
- [问题类型太多导致页面复杂] → 首版只保留四类闭环状态，不扩展整改时限、责任单位、附件逐项绑定。
