## Context

当前开工条件试点的页面已经能展示多轮 run、人工复核结果、整改闭环对照和发起下一轮复审动作，但两个关键体验问题仍未收口：

1. 状态是双轨的：`checkItem.verdict` 和 `humanReviewQueue.status` 会同时出现在报告页，导致“待人工判断”与“人工驳回”并存。
2. 入口是双轨的：报告页能发起下一轮，资料接入页也能直接重新上传，操作者很难判断哪一个才是正式入口。

这轮设计聚焦“最终状态归一”和“唯一复审入口引导”，不扩展数据库、权限或对象建模。

## Goals / Non-Goals

**Goals:**
- 为核查项定义唯一最终交付状态。
- 让报告页、整改闭环页使用这套最终状态。
- 让“开始下一轮整改复审”成为 archived run 下的唯一主入口。
- 让资料接入页只在明确复审意图下进入新 run 创建模式。

**Non-Goals:**
- 不新增后端持久化表或复杂状态机迁移。
- 不做责任人/时限驱动，这会放到下一阶段。
- 不处理智能体资产、知识库资产的正式建模，只先在文档中沉淀方向。

## Decisions

### 1. 以前端派生方式实现最终状态归一

优先从现有 `checkItems` 与 `humanReviewQueue` 派生最终交付状态，而不是先改后端状态模型。这样可以最快解决试点交互冲突，并保持改动范围可控。

### 2. 人工复核结果优先于原始 verdict

如果一个核查项存在人工复核结果：

- `confirmed` / `corrected`：视为已完成人工处理，不再作为待整改问题项显示
- `rejected`：显示为“人工驳回，需补件整改”
- `deferred` / `open`：显示为“待人工处理”

这能避免“needs_human_review` 继续压过人工处理结果。

### 3. archived run 下资料接入页默认只做承接，不直接开放复审创建

只有通过“发起下一轮整改复审”进入时，资料接入页才进入复审上传模式。这样可以把“我要不要开始新一轮”这个决定留在报告页，而不是在多个页面重复出现。

## Risks / Trade-offs

- [前端派生和后端状态可能暂时不完全一致] → 当前先以交付展示一致性为优先，后续如需要可把归一结果下沉到后端。
- [用户直接从左侧导航进入资料接入页会看到受限提示] → 这是有意的收口，避免多个入口并列。
- [confirmed/corrected 是否等同于通过存在业务语义假设] → 当前试点先采用“人工已处理即不再作为待整改问题项”规则，后续可细化为更明确的通过/修正状态。
