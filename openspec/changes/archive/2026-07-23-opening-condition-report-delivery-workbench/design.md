## Context

当前项目已经完成试点主链路，也完成了状态收口与责任催办，但报告页仍偏“结果堆叠页”。代码上，[src/productWorkspacePages.tsx](C:/project/nanjin/AI-assisted-review-platform/src/productWorkspacePages.tsx) 已经具备：

- 报告摘要
- 闭环对照
- 不符合项列表
- 决策留痕
- 历史轮次列表

问题不在“有没有这些块”，而在“这些块没有被组织成操作员视角的交付工作台”。当前用户反馈也集中在这一点：历史轮次能点但不够像完整详情、不符合项一眼看不出优先级、下一轮入口虽已收口但仍需要更明确上下文。

## Goals / Non-Goals

**Goals:**
- 让报告页先明确“当前查看的是哪一轮、是不是当前 run、是否只读”。
- 让报告页把交付结论、整改闭环、不符合项和历史轮次组织成更清晰的工作台结构。
- 把问题项按 operator-facing priority 分组，减少试点交付时的扫描负担。
- 保持下一轮整改复审的唯一主入口语义，不在历史轮次详情里误暴露变更操作。

**Non-Goals:**
- 不新增后端报告数据结构。
- 不生成 Word/PDF 正式外发文档。
- 不做完整项目级待办中心或跨项目历史检索。
- 不在本轮处理全量中文文案与编码历史问题，只聚焦本轮涉及区域。

## Decisions

### 1. 报告页继续复用现有后端结果，不新造交付模型

当前 `reportAsset`、`packageDiagnostics`、`humanReviewQueue`、`checkItems` 和历史 `workspaceTasks` 已足够支撑交付工作台。相比新增后端 DTO，本轮优先把现有结果组织得更清楚，先兑现“能被操作员稳定使用”的价值。

### 2. 以“选中轮次”为中心组织页面，而不是把所有模块平铺

报告页的真实主语应当是“当前选中的这一轮”。因此页面结构以 `selectedTask` 为中心，顶部先给轮次上下文和交付摘要，后面再展开闭环对照、问题清单和历史列表。这样历史轮次切换后，页面语义不会散。

### 3. 不符合项按交付优先级分组，而不是继续用单一列表

对试点交付最重要的是先区分：
- 阻塞项
- 不通过待整改项
- 待人工判断项
- 一般提示项

这比单纯罗列每条 finding 更接近成熟平台的交付页面，也更利于监理快速判断“下一轮补什么”。

### 4. 下一轮整改复审按钮只跟随当前 archived run

历史轮次详情仍保留只读快照和隐藏测试轮次等清理动作，但“发起下一轮整改复审”只在“当前 run 且已归档”时出现，避免历史轮次详情误导成可发起入口。

## Risks / Trade-offs

- [前端结构更丰富，页面体量增加] -> 通过明确的摘要区、分组区和历史区保持信息层级，不增加新的后端复杂度。
- [仍复用现有前端派生数据，可能存在局部文案/编码历史问题] -> 本轮只在报告工作台涉及区域做窄改，避免全文件大重写。
- [不改后端意味着正式导出文档仍未补齐] -> 在 spec 中明确这轮只解决工作台交付可读性，不解决外发文档生成。
