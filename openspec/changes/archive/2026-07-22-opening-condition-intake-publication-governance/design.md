## Context

当前“依据与主数据”页已经能展示后端返回的 basis records、master-data records 和知识库列表，但整体仍停留在调试式视图：

1. 当前 run 到底消费了哪一版依据、哪些主数据，缺少清晰快照。
2. 待确认、待发布、已发布、已驳回、已过期等记录混在一起，操作者无法快速判断下一步。
3. 页面仍直接暴露底层状态枚举，理解成本高，不利于试点交付。

本轮不是增加新接口，而是先把已有结构转成“可运营的发布治理面”，让 run 的前置资产治理可见、可解释、可复用。

## Goals / Non-Goals

**Goals:**
- 让操作者在一个页面内看清当前 run 绑定的依据版本、主数据事实和知识库上下文。
- 让待确认、待发布、已发布和异常记录分区显示，明确下一动作。
- 统一 basis/master-data 的状态文案与展示方式，避免散落的枚举直出。
- 保持与当前试点后端契约兼容，不新增 API。

**Non-Goals:**
- 不在本轮新增数据库迁移、后端持久化表或新的发布审批接口。
- 不在本轮解决完整多项目权限、多租户隔离或责任流派单。
- 不继续扩展报告文档导出格式。
- 不把知识库建设流程做成独立工作流编排器。

## Decisions

### 1. 新增独立能力 `opening-condition-publication-governance`

虽然页面主要落在“依据与主数据”页，但这次不是普通 UI 调整，而是把“哪些资产已可供正式核查消费”变成独立能力。这样后续不管页面怎么改，发布治理的业务约束都能独立沉淀。

### 2. 采用“四层治理视图”而不是继续堆原始列表

页面按以下层次组织：

1. 发布门禁摘要
2. 当前 run 绑定快照
3. 待确认 / 待发布队列
4. 已发布目录与异常记录

原因：
- 操作者第一眼先看 run 是否可用。
- 第二眼看当前 run 实际绑定了什么。
- 第三眼才处理发布动作。
- 历史异常单独收口，避免污染主作业面。

### 3. 前端派生治理分组，优先复用现有字段

后端已有足够字段支持治理分组：

- basis `status`: `draft | pending_confirmation | confirmed | published | superseded | rejected`
- master data `status`: `provisional | confirmed | published | human_approved | rejected | expired`

因此本轮在前端通过 helper 统一映射：
- 操作者标签
- 分组归属
- 提示文案
- 状态摘要

这样能最快交付，也避免为了页面治理先动接口。

### 4. 当前 run 绑定快照优先使用“当前 run 事实”而不是只看全量目录

页面中最重要的不是“库里总共有多少条”，而是“本次 run 真正绑定并消费了哪些依据/主数据”。因此快照区优先展示：

- 当前 run basis 版本
- 当前 run master data facts
- 当前 run bound knowledge base

目录区再展示已发布资产池与异常记录。

### 5. 统一状态文案，不在页面直接显示底层 enum

新增 basis/master-data 状态标签与说明 helper，页面展示转成中文业务语义，例如：

- `pending_confirmation` -> `待人工确认`
- `confirmed` -> `待发布`
- `human_approved` -> `已人工确认`
- `superseded` -> `已被新版本替代`

原因：
- 降低操作者理解成本。
- 避免后续多个页面各自翻译状态导致不一致。

## Risks / Trade-offs

- [前端分组逻辑变多] -> 通过领域 helper 集中派生，而不是把判断散落在 JSX 中。
- [basis 与 master-data 状态语义不完全一致] -> 用统一的治理分区承载，但保留各自的状态标签和说明。
- [本轮不新增后端接口] -> 某些“发布历史”只能通过现有字段推断，先满足试点使用，后续再补正式审计结构。
- [页面信息密度上升] -> 采用分区摘要、目录块和异常区收口，不回退到营销式卡片布局。
