## Why

当前开工条件试点已经能完成资料上传、正式核查、人工复核和报告归档，但“依据与主数据”仍主要以原始记录列表展示。操作者还看不清当前 run 究竟绑定了哪一版依据、哪些主数据仍待确认、哪些记录已经可以发布、哪些记录已被驳回或失效，这会直接影响试点可重复运行和正式交付可信度。

下一阶段最值得投入的，不是继续局部打磨报告页，而是把“入库预览与发布治理”做成清晰的运营面。只有先把识别结果如何确认、何时发布、当前 run 消费哪版资产讲清楚，真实资料试点才能稳定复用。

## What Changes

- 将依据与主数据页从原始记录列表升级为“发布治理面”，显式区分当前 run 绑定快照、待确认/待发布队列、已发布目录和异常记录。
- 为依据记录补充“待确认 / 待发布 / 已发布 / 已替代 / 已驳回”分组展示，为主数据记录补充“待确认 / 人工确认 / 已发布 / 已驳回 / 已过期”分组展示。
- 在页面中突出当前工作区、审查对象、参建主体上下文，并明确当前 run 消费的是哪版依据、哪些主数据事实。
- 将原始状态枚举转换为操作者可理解的业务文案，减少直接暴露底层 enum 带来的理解成本。
- 保持本轮为前端治理面增强，优先复用现有后端字段与接口，不新增新的后端 API。

## Capabilities

### New Capabilities
- `opening-condition-publication-governance`: 定义依据与主数据的发布治理视图，包括当前 run 绑定快照、待发布队列、已发布目录和异常记录。

### Modified Capabilities
- `opening-condition-intake-preview-and-publish-gate`: 预览门禁不再只显示“能不能过”，还要明确展示当前 run 正在消费的发布资产与待处理发布动作。
- `opening-condition-basis-workflow`: 依据工作流需要增加面向操作者的发布队列、已发布版本和替代/驳回记录可见性。
- `opening-condition-master-data`: 主数据工作流需要增加待确认、人工确认、已发布与异常记录的治理展示与当前 run 绑定视图。

## Impact

- `src/productWorkspacePages.tsx`：重构依据与主数据页为发布治理面。
- `src/domain/openingConditionReview.ts`：补充状态标签、分组与页面辅助派生逻辑。
- `src/styles/opening-condition.css`：补充治理面分区、目录列表、状态摘要与异常记录样式。
- `openspec/specs/*`：同步依据、主数据和预览门禁的行为要求。
