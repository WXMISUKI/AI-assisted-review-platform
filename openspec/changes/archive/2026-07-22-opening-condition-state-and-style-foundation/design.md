## Context

当前项目已经有 `DESIGN.md`、主题 token 和较完整的开工条件工作台，但落地方式出现了两个明显问题：

1. 状态治理分散：
   - archived 只读
   - rerun 上传模式
   - 执行台可变更动作
   - 报告页历史/当前轮边界
   这些判断分散在组件内部，容易出现“某处修了、另一处漏了”的现象。

2. 样式治理分散：
   - `src/styles.css` 已超过 4000 行
   - token、壳层、业务样式混在同一文件
   - 开工条件工作台相关样式跨多个位置分布

用户已经明确感受到：这样会影响后续整个平台的持续开发效率，因此这轮应先把治理基础立住。

## Goals / Non-Goals

**Goals:**

- 为开工条件工作台建立共享 UI 派生状态模型。
- 把 archived/current/rerun 的前端边界收口成单一来源。
- 将 `styles.css` 拆出 token 层和开工条件业务层样式文件。
- 保持当前功能不倒退的前提下，让后续开发有可遵循的结构。

**Non-Goals:**

- 不引入新的全局状态库。
- 不在这轮重写整个产品门户和施工方案审查门户。
- 不做全面视觉重设计，只做样式组织治理和少量共享样式抽离。
- 不改后端状态机或持久化模型。

## Decisions

### 1. 采用“共享派生状态模块”而不是直接上新的全局状态管理库

当前问题不是跨页面异步状态爆炸，而是 UI 边界判断散落。最小且高价值的做法是先把这些边界收口成一个共享派生模块，统一输出：

- 是否 archived
- 是否只读
- 是否允许上传新 run
- 是否允许变更当前 run

后续如果页面继续扩展，再考虑更重的状态管理方案。

### 2. 样式先按“token / feature”拆分，而不是一次性全面组件化

`styles.css` 已经过大，但直接迁移到 CSS Modules 或重建组件库会打断主线。当前更合适的做法是：

- `theme-tokens.css`：主题变量与根 token
- `opening-condition.css`：开工条件工作台业务样式
- `styles.css`：仍作为统一入口，承接通用壳层和导入关系

这能先降低管理复杂度，又不需要改动现有 Vite/CSS 组织方式。

### 3. 先把“开工条件工作台”作为治理样板

用户关心的是整个平台长期可维护性，但最有价值的起点仍然是当前正在高频推进的工作台。先在这一条线上形成模式，再复制到其他业务门户。

## Risks / Trade-offs

- [仍保留全局 CSS 入口] → 这是有意为之，先降低重构风险，再逐步继续拆层。
- [共享派生状态仍然是前端层规则] → 当前目的就是先消灭散落布尔判断，后续再决定是否下沉更深抽象。
- [业务样式拆分后仍有部分通用类残留在主文件] → 允许第一轮不求一步到位，但要建立清晰迁移方向。
