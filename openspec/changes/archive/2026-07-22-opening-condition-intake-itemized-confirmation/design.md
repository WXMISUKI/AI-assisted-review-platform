## Context

我们已经补齐了两层基础：

1. 资料接入页的“识别预览确认工作台”
2. 依据与主数据页的“发布治理面”

但当前操作仍然偏粗：
- basis 只能按当前 run 整体发布
- master data 只能按当前 run 批量确认

这对试点有价值，但离真实投产还差一层。真实监理不会总是“一键全过”，而是会：
- 某一条候选通过
- 某一条候选驳回
- 某一条候选留备注说明为什么通过/驳回

本轮目标就是把这层最小粒度补上。

## Goals / Non-Goals

**Goals:**
- 支持 basis 候选发布前附加备注。
- 支持 master-data 候选逐条 approve / reject。
- 让逐条动作发生在资料接入页的候选预览区，而不是要求用户跳到别处。
- 保持与现有后端 API 契约一致。

**Non-Goals:**
- 不做逐字段编辑器。
- 不新增 basis reject 专用后端接口。
- 不做多级审批流。
- 不处理历史版本差异分析。

## Decisions

### 1. 逐条动作优先挂在识别预览确认工作台

用户此时的心智是“系统刚识别出来这些候选，我现在决定哪些能正式入库”。所以逐条动作必须离候选预览最近，而不是埋在治理页或别的详情页。

### 2. basis 采用“备注 + 发布”，master-data 采用“逐条 approve / reject + 备注”

当前后端能力决定了本轮最佳落点：
- basis：使用 `upsert + publish`
- master data：使用现有 `decide(... approve|reject, safeNote)`

这条实现路径与现有契约完全对齐，风险最小。

### 3. 保留批量按钮作为快速路径

逐条确认是为了真实投产，不是替代快速演示路径。因此：
- 执行台与 Trial Intake Overview 里的批量动作继续保留
- 资料接入页增加更细粒度的候选确认面

这样兼顾效率与精细化。

## Risks / Trade-offs

- [basis 仍缺少独立 reject 接口] -> 本轮先不补 reject，只做备注后发布，避免超出现有后端能力。
- [页面动作变多] -> 把逐条动作只收口在候选预览工作台，不扩散到多个页面。
- [备注是安全备注，不是完整审计体系] -> 先满足真实试点确认需要，后续再沉淀正式审计链。
