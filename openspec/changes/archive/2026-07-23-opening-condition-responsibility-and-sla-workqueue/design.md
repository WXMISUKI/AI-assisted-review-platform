## Context

当前代码里已经存在 `deriveOpeningConditionRunActionOwnership`，并且资料接入页、人工复核页、报告页也能显示责任摘要。这说明“责任归属”这个方向已经起步了，但还没有形成真正的工作台能力。

现状问题主要有三类：

1. 责任摘要存在，但缺少“下一步去哪里”的明确路由语义。
2. 工作台概览页仍然偏静态，不能作为操作员的第一落点。
3. 人工复核页能列项，但没有把待处理优先级、延期状态和责任移交做成更强的流程表达。

因此这轮不做新的后端状态，而是把已有状态派生成一个更完整的 operator-facing workqueue 模型。

## Goals / Non-Goals

**Goals:**
- 扩展共享责任派生，补充推荐页面、主动作标签和阻塞计数语义。
- 在工作台概览页落一个统一的责任与时效看板。
- 在人工复核页把 open / deferred / resolved 队列分层展示，并表达责任何时移交到报告交付。
- 保持所有动作仍然围绕现有页面入口，不增加新的状态机分支。

**Non-Goals:**
- 不新增后端 SLA 持久化字段。
- 不做消息通知、邮件提醒或定时任务。
- 不做跨工作区汇总待办中心。
- 不做多角色权限细分。

## Decisions

### 1. 继续以前端共享派生模型为中心

当前试点最值钱的是把工作流语义做清楚，而不是立刻扩后端字段。已有 `pilotTask + readiness + humanReviewQueue` 足以推导责任、时效和推荐页面，因此本轮继续使用共享前端派生模块。

### 2. 推荐页面作为责任摘要的一部分

“下一动作”如果没有页面落点，操作员仍然要自己猜去哪里。于是本轮把 `recommendedPage` 和 `primaryActionLabel` 一起纳入责任派生结果，让概览页可以直接给出明确跳转。

### 3. 人工复核按处理优先级而非原始列表展示

对操作者来说，真正重要的是：
- 还有多少待处理
- 哪些是延期但仍阻塞
- 哪些已经闭环

因此人工复核页应先按 `open / deferred / resolved` 三组组织，而不是沿用原始数组顺序。

## Risks / Trade-offs

- [仅做前端派生，时效提醒不是强 SLA] -> 文案明确为“试点窗口”和“催办提示”，避免误导为正式考核时限。
- [概览页信息密度增加] -> 采用摘要卡 + 单一主动作按钮，不在概览页堆叠细项列表。
- [推荐页面是启发式规则] -> 先基于当前 run 状态做稳定映射，后续如对象建模扩展再继续细化。
