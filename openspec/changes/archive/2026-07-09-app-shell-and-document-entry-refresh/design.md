## Context

当前平台的登录后体验已经具备企业审查工作台雏形，但入口层还没有完全收紧：用户区、主题切换和退出动作需要像全局能力一样常驻；上传入口要同时支持拖拽与文件选择；文档详情要从库内嵌套工作区转成独立页面，减少“页面里再套一层页面”的感觉。

## Goals / Non-Goals

**Goals:**
- 让 shell 里的用户区和主题切换成为始终可见的全局控件。
- 让文档库入口支持拖拽和点击上传，并允许可选元信息。
- 让文档详情以全屏工作页承接，回退动作明确。
- 让主题系统继续以语义 token 为中心，避免组件级写死颜色。

**Non-Goals:**
- 不在本 change 中接入真实文件存储、权限系统或后端任务队列。
- 不在本 change 中重做审查工作台的划词浮层和标注交互。
- 不在本 change 中实现知识库或数据资产的真实管理能力。

## Decisions

### 1. 全局用户区固定存在
用户身份、角色、主题切换和退出动作应当在所有 shell 页面中保持固定可见，不依赖当前选中的导航项。这样可以避免用户在切换页面后找不到全局操作，也更符合成熟的工作台产品习惯。

### 2. 主题采用 token + CSS 变量
浅色主题为默认，深色主题作为可切换状态。所有背景、边框、文本、状态色都通过语义 token 承载，组件层尽量不要自己定义一套新颜色，防止出现“深色主题里又冒出白卡片”的问题。

### 3. 详情页采用独立工作页
文档详情不是库页里的子面板，而是一个单独的全屏工作页。这样既方便承载复杂审查工作流，也为后续真正的路由化迁移留出余地。

### 4. 上传入口统一成一个输入面
拖拽区、文件选择按钮、文件名、项目名称都属于同一个上传入口。文件名和项目名称默认可空，系统用文件名和“未知项目”兜底，降低录入门槛。

## Risks / Trade-offs

- [风险] 全屏详情页会让页面状态切换更明显。→ [缓解] 保持返回按钮显著可见，并沿用当前 mock 状态流。
- [风险] 深浅主题统一后，局部组件可能显得平。→ [缓解] 通过边框、层级和状态色而不是大面积黑底来做区分。
