## Context

我们已经验证了平台可以从真实资料进入正式核查、人工复核和报告归档，但当前链路仍更像一次性 demo，而不是可投产的工程资料审核流。真正的业务闭环通常是：

1. 施工/分包第一次报审；
2. 平台输出不符合项报告；
3. 施工单位补件或整改；
4. 监理发起第二轮复审；
5. 保留第一轮与第二轮差异，最终形成历史台账。

现有系统已经具备 run 粒度与 archived 不可变规则，这为“多轮整改复审 + 历史留痕”提供了自然基础。

## Goals / Non-Goals

**Goals:**
- 让报告页成为可交付的内部辅助意见页，而不是只显示统计摘要。
- 让同一工作区支持多轮核查与整改复审，并保持历史 run 不可变。
- 让操作员能查看历史报告、处理摘要与轮次差异。
- 建立安全的历史处置策略：默认保留，有限软删除。

**Non-Goals:**
- 不在这一轮做正式 docx 生成器。
- 不在这一轮做完整 RBAC、租户隔离与数据库大迁移。
- 不深挖 OCR/LLM 命中率优化，也不扩展到现场核查全域。

## Decisions

### 1. 以“run”为复审单位，而不是在同一个 task 上反复覆盖

原因：这最符合现有 archived 不可变规则，也最接近成熟平台的 revision/version 模型。每一轮 run 都应保留当时的输入、结论、人工决策和报告。

备选：
- 复用同一个 task 覆盖旧结果：实现看似简单，但会破坏审计链。
- 另建完全独立模块：边界过重，不适合试点阶段。

### 2. 报告页优先输出结构化 findings，而不是先做导出文件

原因：当前最大价值是让监理在页面内看懂结果并推动复审。先把 findings 结构做对，后续 docx/pdf 导出才有稳定输入。

### 3. 删除策略采用软删除/隐藏而非默认硬删除

原因：工程审查平台天然偏审计留痕。正式 run 的默认行为应是归档保留；删除更多用于测试数据或误建任务清理。

## Risks / Trade-offs

- [历史 run 增多后界面会变复杂] → 先只做当前工作区的 run 列表和简要对比，不扩展复杂筛选。
- [报告 findings 结构可能先比较“模板化”] → 先保证字段完整、可读，再逐步优化表达质量。
- [软删除语义容易和归档混淆] → 明确区分：归档是正式保留完成记录；软删除是管理员隐藏误建/测试记录。

## Migration Plan

1. 先补 OpenSpec，明确 findings delivery 与 rerun history 的边界。
2. 优先实现报告页 findings 化和 run 历史列表。
3. 再补复审发起、历史切换与软删除。
4. 最后视试点反馈决定是否推进 docx 导出和前后轮对比增强。

## Open Questions

- 历史列表是否需要“按危大工程对象”进一步分组，而不只是按 workspace。
- 报告 findings 的风险等级与整改时限，是先规则化模板输出，还是允许人工编辑后归档。
