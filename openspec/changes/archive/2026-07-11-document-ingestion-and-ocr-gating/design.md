## Context

当前平台已经具备上传、MinIO 存储、OCR 作业提交、任务列表和审查工作台的雏形，但任务生命周期仍然偏“上传后即进入审查”的 demo 逻辑。用户希望看到的是更贴近成熟审查平台的流程：先 OCR 识别、再结构化整理、最后进入审查智能体的流式问题生成。

本次设计的重点不是增加新的 AI 能力，而是把“文档接入阶段”和“审查阶段”分开，让任务状态、页面解锁条件和后端 OCR 作业之间形成稳定的闸门关系。

## Goals / Non-Goals

**Goals:**
- 明确 OCR 识别是审查前置阶段，而不是审查本身。
- 让文档库可见 OCR 进度，用户知道为什么暂时不能进入工作台。
- OCR 完成后，再自动进入审查准备或审查处理中状态。
- 保持任务 aggregate 和前端工作流可被后端 API 替换。

**Non-Goals:**
- 不实现真实 OCR Markdown 重建、表格识别细化或版面纠错算法。
- 不实现真正的多轮 LLM 结构化整理和审查推理。
- 不引入 WebSocket、后台队列、分布式 worker。
- 不改动当前审查工作台的核心交互，只调整进入时机与状态门禁。

## Decisions

1. **把 OCR 作为审查前置闸门**
   - 选择原因：成熟平台一般先做结构化解析，再做审查。这样段落、页码、锚点和问题定位更稳定。
   - 备选方案：上传后立即打开审查工作台，OCR 在后台慢慢补。这样交互更“快”，但用户会在不稳定内容上做审查，后续回流风险很大。

2. **继续沿用单一任务 aggregate，但增加阶段语义**
   - 选择原因：当前任务已经承载文档元数据、OCR 作业和审查结果。我们不需要立即拆成多张表，只需要让阶段语义更清晰。
   - 备选方案：拆成“接入任务”和“审查任务”两个对象。未来可扩展，但对当前 MVP 来说过重，会增加同步和恢复复杂度。

3. **OCR 完成后再启动审查智能体流式工作台**
   - 选择原因：这样可以把“识别中百分比”和“审查中流式阶段”分成两个清楚的体验，符合用户预期。
   - 备选方案：OCR 完成后直接跳进工作台并在后台静默整理结构。虽然页面切换少，但用户会分不清当前到底是在识别还是审查。

4. **在文档库中直接呈现 OCR 阶段和禁入态**
   - 选择原因：用户最先接触的是文档库，这里应该告诉他们任务处于哪一层，避免重复点开详情页。
   - 备选方案：把状态都藏到详情页。这样实现简单，但会让文档库失去“任务控制台”的意义。

## Risks / Trade-offs

- [Risk] OCR 和审查被拆成两段后，任务状态会更复杂。→ [Mitigation] 继续复用同一个 task aggregate，前端只增加阶段标签和进入条件。
- [Risk] 用户可能误以为 OCR 完成就等于审查完成。→ [Mitigation] 明确展示“识别完成 / 审查准备中 / 审查处理中”等不同状态。
- [Risk] 当前没有真实 OCR 内容回流，审查流式内容仍是 mock。→ [Mitigation] 将本次 change 限定为状态门禁和流转，不把真实解析结果纳入范围。
- [Risk] 如果后端 OCR 进度接口不稳定，UI 会频繁闪动。→ [Mitigation] 保持轮询节流，并对失败态提供明确可恢复提示。

## Migration Plan

1. 先把任务状态和 OCR 进度字段补齐。
2. 文档库展示 OCR 进度与禁入态。
3. 详情页从上传态改为 OCR 完成后才可进入。
4. OCR 完成后再启动审查智能体的流式加载页。
5. 最后再把真实 OCR 结果回流接进文档内容渲染。

## Open Questions

- OCR 进度百分比应该来自服务端 `extractProgress`，还是前端做平滑插值？
- 识别完成后，是否自动进入审查加载页，还是要求用户手动点击“开始审查”？
- 审查准备阶段的 loading 页是否需要显示“文档结构化整理中”的中间态？
