## Context

最近几轮变更已经把开工条件试点工作台补到了一个可运行的基础面：

1. 状态归一与整改入口收口
2. 资料接入预览确认
3. 依据与主数据发布治理
4. 历史轮次对比与报告闭环

但这些能力目前仍然更偏“单轮视角”。一旦进入整改复审，页面虽然已经能保留历史 run，也能新建下一轮，但当前 run 的依据、主数据和知识库与上一归档轮次之间缺少一个明确的“复用/差异”解释层。

现状带来的问题：

- 操作者不知道当前 run 绑定的 basis / master data 是上一轮延续来的，还是本轮新识别的。
- 当前 run 即便复用了已确认资产，页面也仍像一组新的候选，容易重复确认。
- 治理页和资料接入页虽然都能看当前 run，但没有统一地表达“复审到底变了什么”。

本轮要补的是这一层“复用/差异解释模型”，而不是重做后端状态机。

## Goals / Non-Goals

**Goals:**
- 为当前 run 派生出基于上一归档轮次的 basis / master-data 复用与差异视图。
- 在资料接入页展示“上一轮可复用资产”“本轮新增/待确认项”，减少重复确认。
- 在治理页展示同一套差异语义，保证跨页一致。
- 保持 archived run 只读，所有复用判断仅用于说明和引导当前 run。

**Non-Goals:**
- 不新增数据库迁移或后端持久化字段。
- 不改变后端 intake / publish / master-data decision API 契约。
- 不在本轮实现自动跳过确认的后端强制逻辑，只先把操作语义和视图做清楚。
- 不扩展到报告文档导出格式重做。

## Decisions

### 1. 先做前端派生复用模型，不改后端契约

当前后端已经能提供：
- 当前 run 绑定的 basisVersion
- 当前 run requiredMasterData
- 工作区历史 runs
- 记录状态（published / human_approved / confirmed / rejected 等）

因此本轮不值得先扩后端字段，而是先在前端用“当前 run + 上一归档 run”派生出可复用视图，快速验证产品语义。

备选方案是增加专门的后端 reuse diagnostics 字段，但会把一次主要是工作台产品化的问题提前做成 API 扩展，收益不如先前端验证。

### 2. 差异判断以“上一归档轮次”为比较基准

整改复审的核心是和上一轮相比有没有补正，所以比较对象应当是：
- 当前选中的 current run
- 同工作区时间上最近的 previous archived run

而不是全工作区任意旧记录，也不是整个目录的最新 published 资产。这样最贴近监理的真实工作心智。

### 3. 资产差异只分操作者需要关心的几类

本轮不引入复杂枚举，而是收敛成以下几类：
- `reused`: 本轮仍沿用上一轮已确认资产
- `new_for_current_run`: 本轮新增绑定的资产
- `needs_reconfirmation`: 本轮资产存在待发布/待确认状态
- `dropped_from_current_run`: 上一轮在用、但本轮已不再绑定的资产

这四类足够支撑“我要不要再确认”和“本轮到底改了什么”两个核心问题。

### 4. 展示重点放在资料接入页和治理页，历史页只保持只读对照

资料接入页是操作者处理新一轮补件的主场景，治理页是核对目录资产的主场景，所以复用/差异必须同时出现在这两个页面。

报告页已经承担了 findings 和整改闭环，不在本轮继续堆叠资产差异视图，避免页面职责混乱。

## Risks / Trade-offs

- [只做前端派生，缺少后端显式 reuse 标记] -> 先以当前/上一轮真实数据派生，等试点证明有价值后再决定是否下沉为后端字段。
- [同 ID 复用不一定等于内容完全不变] -> 本轮只表达“平台当前绑定关系”的复用语义，不声称做内容级 diff；文案上避免过度承诺。
- [页面信息密度继续上升] -> 使用紧凑分组卡片和摘要统计，只把最关键的复用/新增/待确认信息前置。
