## Why

当前平台已经能跑通“上传 -> 识别预览 -> 正式核查 -> 人工复核 -> 报告归档”，但一进入整改复审场景，操作者仍然缺少最关键的信息：上一轮已经确认过的依据、主数据和知识库，到底哪些可以直接复用，哪些是本轮新识别的，哪些虽然沿用了旧资产却仍需要重新确认。结果就是每一轮都像重新开一遍空白任务，既增加重复操作，也削弱了版本化复审的价值。

从快速投产角度看，下一阶段最值得补的不是继续打磨局部样式，而是把“复审复用与差异处理”做成操作员看得懂、点得清、不会误重复确认的工作台能力。这样平台才能真正从“能跑一轮”进化到“能稳定跑多轮整改复审”。

## What Changes

- 为整改复审 run 增加“资产复用与差异”语义，区分当前 run 中的复用资产、新增资产、待重新确认资产和上一轮已退出使用的资产。
- 在资料接入页的识别预览与 intake overview 中，展示“上一轮可复用资产”和“本轮变化项”，让操作者知道哪些不用重复确认。
- 在依据与主数据治理页中，补充当前 run 相对上一归档轮次的复用快照和差异清单，保持与资料接入页一致的状态语义。
- 保持历史 run 只读，不回写旧轮次；所有复用判断只服务于当前 run 的确认与发布动作。

## Capabilities

### New Capabilities
- `opening-condition-rerun-asset-reuse`: 定义整改复审 run 如何继承上一轮已确认资产，并向操作者展示复用、新增与待重确认差异。

### Modified Capabilities
- `opening-condition-intake-candidate-preview`: 识别预览需要区分当前 run 的复用资产和本轮新增/待确认资产，而不只展示平铺候选。
- `opening-condition-publication-governance`: 治理页需要展示当前 run 相对上一归档轮次的复用快照与差异，而不只展示静态目录。
- `opening-condition-rectification-rerun-history`: 新的整改复审 run 除了保留历史，还需要明确说明它继承了哪些上一轮已确认资产。

## Impact

- `src/productWorkspacePages.tsx`：新增复审复用/差异的前端派生视图，并在资料接入页、治理页中落地展示。
- `src/App.tsx`：继续沿用现有 run 解析与刷新逻辑，不引入新的 run 变更入口，只消费当前与历史 run 数据。
- `src/styles/opening-condition.css`：补充复用资产卡片、差异分组和紧凑状态标签样式。
- `openspec/specs/*`：同步复审复用与差异语义，避免页面文案和规格定义脱节。
