## Why

OCR 水合和结构恢复已经打通，审查准备 loading 也开始吃真实 recovered structure，但工作台里的初始问题仍然主要依赖固定 mock issue 套件。这样一来，任务在结构层面已经“真实”，在问题层面却还是“演示”，审查流最后一段会显得断裂。

我们需要让恢复后的章节和段落能够自动补出一批稳定的结构化问题草案。它们不是真实 LLM 审查结果，但要足够像真实审查：有明确锚点、可读理由、可追溯依据，并且能和现有问题列表、人工标注和完成流程协同工作。

## What Changes

- 为 recovered structure 新增一层确定性的草案问题生成逻辑，基于段落文本和章节语义匹配常见审查风险。
- 在 OCR 水合成功后，将草案问题与现有问题合并，并保持手工问题和已有问题不丢失。
- 让文档任务、会话服务和本地持久化都理解“结构驱动问题草案”这一中间层。
- 保持没有匹配到草案时的安全回退，不阻塞工作台。

## Non-goals

- 不接入真实 LLM 问题生成。
- 不做复杂规则库、向量检索或外部知识库匹配。
- 不改变问题卡片的 UI 结构或完成报告格式。

## Impact

- `src/domain/`: 新增结构化问题草案生成与合并逻辑。
- `src/domain/reviewSessionService.ts`: OCR 水合和会话创建时使用结构驱动草案。
- `src/domain/reviewTaskRepository.ts`: 本地任务恢复时自动补齐结构驱动问题草案。
- `src/domain/mockReviewTaskSeeds.ts`: 种子任务可复用同一套草案生成逻辑，保持演示数据与真实结构一致。
