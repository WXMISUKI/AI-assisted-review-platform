## Context

当前平台已经具备：

- 真实资料接入
- 正式核查
- 人工复核
- 报告生成与归档
- 多轮整改复审历史

但操作者视角仍缺少一套稳定的作业驱动语义：

1. archived 轮次虽然整体改成了承接页，但局部按钮仍可能漏出可点动作。
2. 页面能展示状态，却不能清楚说明“现在该谁处理、下一步做什么、是否已经拖期”。
3. 当前试点还是单项目单工作区，但已经需要具备作业台账产品的最小表达能力。

这轮设计不做后端状态机重构，而是先把责任与动作语义前端派生出来，验证试点操作是否顺手。

## Goals / Non-Goals

**Goals:**

- 补齐 archived 轮次在资料接入页的只读一致性。
- 为当前 run 派生当前责任人、下一动作、到期状态。
- 让执行台、人工复核页、报告归档页、历史详情页都能使用同一套动作语义。
- 用最小代价验证“责任人与下一动作驱动”是否足以支撑试点作业。

**Non-Goals:**

- 不新增后端数据库字段或正式 SLA 表。
- 不做完整的组织、项目、危大工程、分包对象建模。
- 不在这轮建设多角色消息提醒、通知中心或自动催办。
- 不扩展智能体资产或知识库资产的正式管理界面。

## Decisions

### 1. 先用前端派生模型表达责任人与下一动作

当前已有足够的输入：

- `pilotTask.state`
- `pilotTask.humanReviewQueue`
- `pilotTask.reportAsset`
- `pilotReadiness.preflightReadiness`

因此这轮先不改后端模型，而是在前端派生：

- `currentOwner`
- `nextAction`
- `dueState`
- `actionReason`

这样可以最快验证试点作业流是否更清晰。

### 2. 用统一谓词控制 archived 只读护栏

只读一致性不应该散落在单个按钮里手工判断。执行台、Trial Intake Overview、真实试点上传面板都应共享同一判断：

- archived run
- 且当前不在 `rectification_rerun` 模式

满足条件时，所有会变更当前 archived run 的动作都必须禁用。

### 3. 时限状态先采用“试点规则表”而不是持久化 SLA

这轮不引入后端 due date 字段，而是按作业阶段定义试点规则：

- `packet_uploaded` / `blocked_*`：提示尽快补齐前置门禁
- `awaiting_human_review`：监理待处理
- `report_ready`：待生成报告
- `archived`：无当前责任人，下一动作转为“如需继续则发起下一轮整改复审”

如需时限展示，则按 `updatedAt + stageDefaultWindow` 前端派生，用于试点验证，不作为正式合同承诺。

### 4. 历史轮次保留动作归属快照，而不是只保留最终结论

成熟平台真正有价值的不只是“这一轮结果是什么”，还包括“那一轮卡在谁手里、为什么卡住”。因此历史详情页应能看到该轮的动作归属语义，便于复盘。

## Risks / Trade-offs

- [前端派生规则与未来后端正式状态机可能不完全一致] → 先在 spec 中固化语义，后续若下沉后端，以当前试点语义为基线。
- [试点时限规则可能过于简化] → 当前只把它当作操作者提示，不作为正式 SLA 承诺。
- [责任人仍是角色级而不是人员级] → 这轮先解决“该哪类人处理”，下一阶段再细化到人员/组织/待办池。
- [同一页新增更多信息块可能增加密度] → 仅展示最小必要信息：责任人、下一动作、时限状态、原因。
